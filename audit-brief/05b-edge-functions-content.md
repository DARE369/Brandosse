# Stage 5b — Edge Functions: Content Planning, Prompt Enhancement, Metadata

## Shared LLM dispatcher — `supabase/functions/_shared/llm.ts`

Not itself an HTTP-triggered edge function, but every function in this stage depends on it, so documenting it once here:

- `callLlm({ systemPrompt, messages, preferredProvider, maxTokens, temperature, jsonMode })` — `resolveProviders(preferredProvider)` normalizes the caller's preference (`"claude"`/`"anthropic"` → Anthropic; anything else → Groq), builds an ordered `[preferred, fallback]` list, **filtering out any provider whose API key env var isn't set** (so if only one key is configured, there is effectively no real failover, just a single-provider call).
- Tries providers **in order**, catching each failure and moving to the next; only throws if all configured providers fail. This means the actual provider/model used for any given call can silently differ from what the client "requested" — the response's `provider`/`model` fields are the only reliable signal of what really served the request, which is exactly why `groqClient.js`'s client-side comment insists callers "must record these, not assume Groq."
- **`callPromptEngine`** (used only by the three media edge functions in Stage 5a) is a separate, narrower helper that always targets Claude Haiku specifically (hardcoded `claude-haiku-4-5-20251001`, 15s timeout via `AbortSignal.timeout`), with a fallback to `callLlm(..., preferredProvider: 'groq')` only if `ANTHROPIC_API_KEY` is entirely unset — i.e., prompt-enhancement's fallback path is Groq, not "no fallback."
- **Correction of a client-side assumption**: `generate-post-metadata` and `generate-caption` both call `callLlm` with `preferredProvider: 'anthropic'` — meaning their primary provider is **Claude**, with Groq only as the fallback if Anthropic's key is missing/the call fails. This is the *opposite* framing from `groqClient.js`'s "Groq primary, Claude fallback" client-side comment, which applies specifically to `generate-content-plan` (that function does call with `preferredProvider: 'groq'`). An architecture reviewer should note that different content-generation edge functions in this same page's flow have **opposite primary/fallback provider orderings** — content plans prefer Groq-first, but post metadata and direct captions prefer Claude-first.

## `generate-content-plan` (`supabase/functions/generate-content-plan/index.ts`)

- **Trigger**: `POST`, invoked by `groqClient.js: callGroqContentPlan`/`callGroqRevision`, once per single-image/carousel generation (`mode: 'plan'`) and again if the quality gate finds a violation (`mode: 'revision'`).
- **Auth**: `requireUser` only — no credit check, no rate limiting visible in this file (a content-plan call itself costs nothing in credits; only the resulting image/video render is billed).
- **Validation**: `mode: 'revision'` requires both `plan` (object) and non-empty `violations` array (400 otherwise); `mode: 'plan'` (default) requires `brief` (object).
- **`generatePlan`**: calls `callLlm` with `preferredProvider: 'groq'`, `jsonMode: true`, `maxTokens: 3500`, `temperature: 0.7`, a very long system prompt embedding the full `ContentPlan` JSON schema skeleton (platforms, carousel slides, visual prompts, caption/title/hashtags per-platform overrides, an SEO sub-object, guardrails check) plus explicit per-platform constraints (X ≤280 chars, Instagram 125-150 words/≤30 hashtags, LinkedIn ≤5-7 hashtags, TikTok 3-5 hashtags, YouTube title ≤70 chars) and carousel slide-count rules. `pickJson` strips markdown code fences or extracts the substring between the first `{` and last `}` before `JSON.parse`, defending against the LLM wrapping its JSON in prose despite instructions.
- **`revisePlan`**: same `callLlm` shape but `temperature: 0.4` (more conservative) and a compliance-editor system prompt instructing the model to fix *only* the listed violations, echoing the full existing plan JSON and the brand summary back to the model as context.
- **Response**: `{ plan, provider, model, totalTokens }` for both modes — the client (`generationPipeline.js`) stores `provider`/`model` onto the `content_plans.plan_provider`/`revision_provider` columns specifically so the DB has an honest record of which LLM actually authored each plan/revision.
- **Error handling**: a single top-level try/catch; a malformed LLM JSON response throws inside `JSON.parse` and propagates as a generic 500-ish error via `mapErrorToStatusCode`/`toErrorPayload` — there is no retry-with-repair loop server-side (the client's own `contentPlanValidator.js` handles *structural* repair after the fact, but a completely unparseable response is a hard failure with no automatic retry).
- **DB writes**: none directly in this function — the `content_plans` insert happens client-side in `generationPipeline.js` after this function returns.

## `enhance-prompt` (`supabase/functions/enhance-prompt/index.ts`)

- **Trigger**: `POST`, invoked by `SessionStore.js: enhancePrompt` action (the Studio "Enhance prompt" button).
- **Auth**: notably **no `requireUser`/auth check at all** in this file — unlike every other edge function in this audit, `enhance-prompt` does not call `createAuthClient`/`requireUser`. It only validates `prompt` is non-empty (400 otherwise via `createHttpError`). This is a real inconsistency worth flagging to a security reviewer: every sibling function in this flow enforces authentication, but this one does not, meaning it could be called anonymously by anyone who knows the endpoint, consuming LLM-provider quota with no per-user attribution or credit cost.
- **Request normalization**: clamps `variantCount` to [1,5] (default 3); truncates `previousPrompts` to 8 entries; normalizes `brandKit` into a compact `{ brand_name, brand_voice, tone, preferred_hashtags, messaging_pillars, do_not_use }` object, returning `null` if none of those fields have content (so an empty/unconfigured brand kit is treated as "no context" rather than sending noise to the LLM).
- **LLM call**: `callLlm` (default `preferredProvider` unset → resolves to Anthropic-first per `resolveProviders`' default `raw = 'anthropic'` fallback when neither the caller nor `DEFAULT_AI_MODEL` env specifies otherwise), `jsonMode: true`, asks for strict `{"suggestions": [...]}"`.
- **Response parsing** (`parseSuggestions`): tries `JSON.parse` first, checking both `suggestions` and `variants` keys; if that fails, falls back to naive line-splitting (stripping leading bullet/number markers) — a second layer of defensive parsing beyond `generate-content-plan`'s `pickJson`.
- **Deterministic fallback**: if the LLM call itself throws (not just returns unparseable content — an actual request failure), `buildFallbackSuggestions` synthesizes N variants by string-concatenating the original prompt with rotating quality adjectives (`"high detail"`, `"clean composition"`, `"strong focal point"`) plus brand voice/tags/avoid-list fragments and a "different from: {last previous prompt}" hint — a crude but always-available fallback that guarantees the endpoint never hard-fails on LLM outages (returns 200 with synthesized suggestions rather than an error).
- **DB writes**: none.
- **Response shape**: `{ enhancedPrompt, suggestions[], provider, model, context_used: { has_brand_context, previous_prompt_count } }` — `provider`/`model` are `null` when the deterministic fallback path was used (since no LLM actually ran).

## `generate-post-metadata` (`supabase/functions/generate-post-metadata/index.ts`)

- **Trigger**: `POST`, invoked by `requestPostMetadataForDraft` (`SessionStore.js`) — after every successful generation's auto-created draft (`ensureDraftForGeneration`), and by `regeneratePostMetadata`.
- **Auth**: `requireUser`; additionally requires `ANTHROPIC_API_KEY` to be configured at all (returns HTTP 500 with an explicit message if not — this function has a **hard dependency on Claude being available**, unlike `enhance-prompt`'s graceful multi-provider fallback).
- **Context loading** (`loadContext`) branches three ways depending on what the request supplies:
  1. **`post_id` provided**: fetches the `posts` row joined with its `generations` row; enforces access — if the post is org-scoped, requires the caller be an active member of that org **and** have access to that specific brand project (`requireActiveOrgMember`/`ensureBrandProjectAccess`); if personal, requires `post.user_id === caller`. Throws (translated to an error response) otherwise — this is the actual server-side authorization boundary for metadata generation, not merely RLS.
  2. **`generation_id` provided** (no post yet — the path used when `ensureDraftForGeneration`'s own insert somehow didn't happen first, or when called with just a generation id): same org/personal ownership check against the `generations` row directly.
  3. **Neither provided**: falls through to using whatever `prompt`/`organization_id`/`brand_project_id`/`media_type` fields were sent raw in the body, with `userId` from the auth token — no ownership check possible in this branch since there's no specific resource to check against (this is a looser path only reachable if a caller invokes the function with a bare prompt, not via any code path traced in this audit).
- **Brand voice sourcing** (`buildBrandPrompt`): for org-scoped requests, builds a system-prompt fragment from the organization's default/specified brand project (`fetchOrganization`, `fetchBrandProject`/`fetchDefaultBrandProject`, `getBrandKitSystemPrompt` with a client-constructed fallback via `buildBrandKitSystemPrompt` if no pre-built org system prompt exists). For **personal** workspace requests, `buildPersonalBrandPrompt` reads the caller's own `brand_kit` row directly (explicitly called out in a code comment as "the headline fix" — implying personal users previously got no brand-voice injection here at all) and constructs "hard rules" (forbidden phrases, max-hashtags) that the LLM is instructed to treat as non-negotiable, separate from softer style guidance.
- **LLM call**: `callLlm({ preferredProvider: 'anthropic', ... })` — Claude-first (see the cross-cutting note above). System prompt combines the brand-voice fragment with fixed metadata-generation rules (title suitable for dashboards/approval queues, 4-8 non-spammy hashtags, video favors a hook-driven caption, YouTube favors an especially clear/searchable title).
- **Post-processing**: `title` falls back to `fallbackTitleFromPrompt` (first 7 words of the prompt) if the LLM omits it or the parse fails to find it; `hashtags` normalized/capped by `brand.maxHashtags` (personal kit's `max_hashtags` column) or a default of 12.
- **DB writes**:
  - If `body.post_id` was supplied: updates only the *requested* `fields` (`title`/`caption`/`hashtags` conditionally, based on `normalizeFields(body.fields)`) plus always updates `workflow_state` (`metadata_status: 'completed'`, `metadata_generated_at`, `metadata_provider`, `metadata_model`) and `updated_at`. **Server-side authoritative marker of completion** — this is what the client's `hydratePostProductionFromGeneration` re-read later checks via `workflow_state.metadata_status`.
  - If a `sessionId` was resolved from context and a `title` was produced: conditionally renames the **session** itself, but only if the session's current title is empty, `'New Session'`, or exactly matches the plain prompt-derived fallback title — i.e., this function can silently overwrite a session's display title with the freshly-generated post title, but only when it looks like the session never got a "real" custom title. This is a cross-cutting side effect a caller invoking this function purely for post metadata might not expect (renaming the *session*, not just the post).
- **Error handling**: single top-level try/catch; a post/generation not found, or an access-check failure, throws a plain `Error` with a human-readable message (not a structured HTTP-coded error via `createHttpError` for those specific checks — relies on the generic `mapErrorToStatusCode` fallback behavior for whatever status that maps a plain `Error` to, likely 500 rather than a more precise 403/404 — UNCLEAR without reading `mapErrorToStatusCode`'s implementation, flagged as worth verifying).
- **No client-side counterpart writes `workflow_state.metadata_status = 'failed'` from a thrown error here** — that responsibility lives entirely in the calling client code (`scheduleDraftMetadataGeneration`'s catch block in `SessionStore.js`), meaning if this edge function's *response itself* is lost in transit (network failure) rather than cleanly erroring, the post could be left indefinitely in `metadata_status: 'in_progress'` (written optimistically by the client before invoking) with no timeout/reconciliation mechanism observed.

## `generate-caption` (`supabase/functions/generate-caption/index.ts`)

- **Trigger**: `POST`, invoked only by `SessionStore.js: generateCaption` — per Stage 4's note, no button in the current `PostProductionPanel.jsx`/`StudioPage.jsx` UI was found calling this store action, so this function appears to be **legacy/superseded** by `generate-post-metadata` for this page's actual current UI, though the code path remains fully wired and functional if invoked.
- **Auth**: `requireUser`; also hard-requires `ANTHROPIC_API_KEY` (500 if missing) — same hard-Claude-dependency pattern as `generate-post-metadata`, but this function has **no fallback to Groq at all** even implicitly, since it doesn't call `callLlm` with any provider-agnostic fallback path beyond what `callLlm` itself does (which would still try Groq as a fallback if Anthropic's *call* fails, but the explicit env-check up front means it never even attempts the call if the key is simply absent).
- **Input**: `imageDescription` (required, 400 if empty), `platform` (default `'instagram'`), `brandKit` (raw object, not the condensed `{summary, raw}` shape other functions expect — `buildBrandContext` here reads `brandKit.summary` directly or falls back to reading individual `raw.*` fields), `previousCaptions` (up to 5), optional `tone` override.
- **Platform character limits** (`PLATFORM_LIMITS`): instagram 2200, twitter/x 280, linkedin 3000 — enforced post-hoc via `limitCaption` (truncates + ellipsis) rather than instructing the LLM to self-limit precisely, so a caption can be abruptly truncated mid-sentence if the model overshoots.
- **LLM call**: `callLlm({ preferredProvider: 'anthropic', jsonMode: true, ... })`, system prompt instructs exactly 5 hashtags, brand-voice/tone/pillars/dont_list adherence, explicit "do not make up facts about the brand."
- **DB writes**: none — this function is purely a stateless caption generator; the client (`SessionStore.js: generateCaption`) is responsible for applying the result to `postProduction` state itself, with no corresponding `posts` table update inside the store action either (a caller would need to separately call `saveDraft`/`updatePostProduction` to persist it).
