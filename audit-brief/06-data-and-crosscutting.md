# Stage 6 — Data & Cross-Cutting Concerns

**Source note**: table/RLS/trigger detail below comes primarily from `supabase/migrations/20260710090000_baseline_core_tables.sql`, which is itself an unusually self-documenting "reconciliation" migration — its own inline comments already flag drift between what migrations claim and what live introspection found (tagged `[LIVE-CONFIRMED]`, `[MIGRATION-TRACKED]`, `[CODE-INFERRED]`, `[FIXED 2026-07-10]`, `[ORIGINAL-SCHEMA]`). Where that file itself says "AMBIGUOUS" or "confirm live," this stage reproduces that uncertainty rather than resolving it — resolving it would require live DB introspection, out of scope here.

## Tables read/written by this page, with relevant columns

### `generations`
- Columns this page's code actually touches: `id`, `user_id`, `session_id` (no enforced FK to `sessions.id` per migration comment — a real, confirmed gap), `prompt`, `enhanced_prompt`, `media_type`, `status` (`queued|processing|completed|failed`, **no CHECK constraint** enforcing this enum at the DB level, unlike `posts.status`), `progress`, `storage_path` (canonical media URL for both images and video), `output_url` (per migration comment, current code always sets this identical to `storage_path` — not used as a distinct field despite the name implying otherwise), `provider`, `provider_model`, `aspect_ratio`, `metadata` (jsonb — carries `lineage`, `edit_mode`, `source_image_url`, `brand_kit_hash`, provider cost/timing fields, carousel slide info), `batch_id`/`batch_index` (used by `generationPipeline.js`'s carousel orchestration but **not tracked by any migration** — confirmed schema drift, the columns exist live but no migration ever added them), `content_plan_id`, `carousel_slide_index`/`carousel_slide_total`/`slide_prompt`, `organization_id`/`brand_project_id`.
- **RLS**: `workspace_scoped_generations_access` (FOR ALL) — `auth.uid() = user_id AND (organization_id IS NULL OR org_current_user_has_brand_access(...))`, or admin. The migration's own comment flags that an older, never-explicitly-dropped policy (`"Users or scoped admins manage own generations"`, from `20260312153000`) may still coexist live alongside this one and the org-scoped SELECT-only policy — i.e., **the actual effective RLS on this table today is unverified from migrations alone**.
- **Triggers**: `generations_audit` (logs to `audit_logs` on UPDATE), `touch_last_active_from_generation_insert`, `zz_sync_generation_org_scope_to_posts` (propagates org/brand-project scope changes onto related `posts`), and critically **`generations_to_draft_post_insert`/`generations_to_draft_post_update`** — a DB trigger (`ensure_draft_post_for_generation()`) that fires on every `generations` insert and on every update where `status` becomes `'completed'`. This means **the "AI output becomes a draft post" hand-off is a DB-level trigger, not solely the client's `ensureDraftForGeneration` helper** — the client-side function of the same conceptual purpose in `SessionStore.js` may be partially redundant with this trigger (both could independently create a draft `posts` row for the same generation; the migration comment explicitly states "no application code decides that a completed generation should become a post," implying the DB trigger is the authoritative mechanism and the client-side `ensureDraftForGeneration`/`scheduleDraftMetadataGeneration` logic reviewed in Stage 3a may be a **duplicate, possibly legacy, client-side reimplementation** of what the DB now already guarantees — worth flagging directly to an architecture reviewer as a likely redundancy/dead-code candidate, not merely a stylistic overlap). No `updated_at`-maintaining trigger exists on this table (every writer, including the client, must set it manually) — unlike `posts`.

### `posts`
- Columns touched: `id`, `user_id`, `generation_id` (required by the draft-creation trigger above), `account_id`, `caption`, `title`, `hashtags` (`text[]`), `platform`, `scheduled_at`, `status` (`draft|scheduled|publishing|published|failed|archived`, **DB-enforced CHECK constraint** — the authoritative enum, matching `POST_STATUS` in `src/constants/statuses.js`), `organization_id`/`brand_project_id`, `pipeline_item_id`, `seo_state` (jsonb, GIN-indexed), `workflow_state` (jsonb, GIN-indexed — carries `metadata_status`/`metadata_error`/`metadata_updated_at`/`seo_status`/`seo_updated_at`), `published_at`/`failed_at`/`error_message`/`external_post_id` (real publish-tracking columns — the migration explicitly warns that a different, earlier-assumed set of column names for this purpose, `platform_post_id`/`platform_post_url`/`failure_reason`/etc. from an unrelated migration, **do not actually exist live**; any code assuming those names would silently fail or no-op).
- **Unique constraint**: `idx_posts_unique_draft_per_generation_account` — one `draft` row per `(user_id, generation_id, account_id-or-sentinel)`. This is the actual DB-level guarantee behind all of `SessionStore.js`'s "find existing draft or insert" reuse logic (`saveDraft`, `publishContent`, `preparePostForApproval`) — a concurrent double-insert attempt for the same generation would hit this constraint rather than silently duplicating, though none of the reviewed client code has an explicit `ON CONFLICT` handler for it (a race between two rapid `saveDraft` calls could surface a raw unique-violation error to the user rather than a friendly message — not verified against actual error-handling code, since none of the reviewed `catch` blocks special-case a Postgres unique-violation code).
- **RLS**: base ownership policy (`"Users or scoped admins manage own posts"`, FOR ALL) layered with `org_workspace_member_read_posts` (SELECT) and `org_workspace_member_schedule_posts` (UPDATE, additionally gated by a `can_schedule` permission check) — the org-scheduling policy is notably **stricter than the generic org read policy**, meaning an org member could see a scheduled org post but not be able to update its `scheduled_at` without the specific `can_schedule` permission.
- **Triggers**: `posts_audit`, `enforce_post_lifecycle` (`lock_terminal_posts()` — sets `is_locked = true` once `published`/`publishing`, **raises an error on illegal status transitions off a terminal status**, and always stamps `updated_at`). This is a real, DB-enforced backstop behind the client-side `assertPostStatusTransition`/`postStatusMachine.js` check — meaning even if the client's own transition-guard logic had a bug or was bypassed, the DB trigger would still reject an invalid transition (e.g., un-publishing a post) at write time. `library_item_after_post_insert` (auto-creates a `content_library_items` row — meaning the client's own `ensureLibraryRowsForPosts` calls throughout `SessionStore.js` may **also** be redundant with this trigger, the same class of client/DB duplication flagged for `generations` above). `set_posts_org_scope_from_generation` (keeps `posts.organization_id`/`brand_project_id` in sync with its generation's — likely overlapping with the client's own `syncOrgScopeToGenerations` call, which actually writes the *other* direction, generation→post scope propagation, so these may not be fully redundant with each other but do both exist as scope-sync mechanisms for the same relationship).

### `sessions`
- Columns touched: `id`, `user_id`, `title`, `metadata` (jsonb — `draft_prompt`, `title_source`), `workspace_type` (`personal|organization`, DB CHECK-enforced), `organization_id`, `brand_project_id`, `project_id` (FK to `studio_projects`), `updated_at` (no maintaining trigger — must be set manually, confirmed asymmetry vs. `posts`).
- **MAJOR FLAG explicitly called out in the migration file itself**: the live `public.sessions` table is **shared with an entirely unrelated tutoring-platform domain** in the same Supabase project/schema — carrying columns like `parent_id`, `tutor_name`, `session_date`, `tutor_notes`, etc., alongside this product's own columns. The migration's own comment calls this "a real architectural risk" (a future `NOT NULL` constraint or trigger added for one product could break the other's inserts) and explicitly defers resolving it to "Phase 7." This is directly relevant to any Generate-page architecture review: `sessions` is not a clean, single-purpose table.
- **RLS**: `workspace_scoped_sessions_access` (FOR ALL), same personal-vs-org branching pattern as `generations`; same caveat about a possibly-still-live older policy not explicitly dropped.

### `content_plans`
- Written once per single-image/carousel generation request by `generationPipeline.js` (`user_id`, `session_id`, `raw_user_input`, `intent_summary`, `content_plan` (full JSON), `groq_model`, `groq_tokens_used`, `plan_provider`, `revision_provider`, `revision_model`, `quality_gate_pass`, `quality_gate_notes`). Not read back by this page anywhere in the traced flow (write-only from Studio's perspective) — its data would presumably be consumed by some analytics/admin surface out of scope here.

### `content_library_items`
- `ensureLibraryRowsForPosts` inserts `{ user_id, post_id, item_type: 'post' }` rows, tolerating missing-table (`42P01`) and duplicate (`23505`) errors as no-ops. Per the `posts` trigger note above, this may be entirely superseded by the DB-level `library_item_after_post_insert` trigger — if so, every one of this page's calls to `ensureLibraryRowsForPosts` is a redundant network round-trip that the trigger has already made unnecessary.

### `user_credits` / `credit_transactions`
- `user_credits`: `balance` (CHECK `>= 0`), `lifetime_purchased`, `lifetime_consumed`. RLS: users can `SELECT` their own row; a policy explicitly named `"Users cannot modify credits directly"` blocks direct `UPDATE` (`USING (false)`) — the **only** legitimate write path is the `deduct_credits` RPC (`SECURITY DEFINER`-style function, not read in this audit) called from the media edge functions. This is a solid defense-in-depth pattern: even if a client somehow tried to `UPDATE user_credits` directly, RLS blocks it outright.
- `credit_transactions`: read by `useCreditSpendByCategory` (category breakdown by `image|video|carousel|edit|other`, with an explicit `"Uncategorized (before tracking)"` bucket for legacy rows predating the category column, per that hook's own code comment referencing a `20260706120000_credit_category_tracking.sql` migration).
- Note the **cross-referenced ambiguity** in the `generations.cost` column: the migration comment states zero code references exist for that column, and current edge functions instead record cost inside `generations.metadata->>'cost_usd'` — a "possible parallel/abandoned convention" per the migration's own flag, meaning any future cost-reporting feature reading `generations.cost` directly would get empty/stale data.

### `studio_projects`
- Read/write via `fetchProjects`/`createProject`/`renameProject`/`deleteProject`/`reorderProjects`. `sessions.project_id` has `ON DELETE SET NULL` (confirmed by the `sessions` table definition's FK clause), backing the "deleting a project re-homes its sessions to General" behavior described in Stage 3b without needing explicit cleanup code.

### `mock_publish_logs`
- Not defined in the reviewed baseline migration excerpt (out of scope to locate its exact DDL in this stage), but its role is confirmed by the `mock-publish` edge function's read/idempotency-check usage (Stage 5c) — stores prior publish-attempt outcomes keyed by `(post_id, connected_account_id, publish_request_id)`.

## Error handling patterns (cross-cutting)

- **Client-side**: every store action follows the same `try { ... } catch (err) { set({ error: err.message }); throw err; } finally { reset transient flags }` shape — errors are always both stored in Zustand `error` state (or a feature-specific status field like `seoStatus: 'failed'`) **and** re-thrown, so the calling component's own `catch` (typically wired to `toast.error`) is the actual point where the user sees anything. A silent-error class exists for background/non-critical operations (`scheduleDraftMetadataGeneration`, `dispatchContentSync`'s consumers, `optimizeSeo().catch(() => {})` in the auto-hydrate effect, session-title generation, session-rename) — these only `console.error`/`console.warn`, with no user-facing surface at all.
- **Server-side (edge functions)**: uniform `handleCors` → method-check → single top-level `try/catch` → `toErrorPayload(error)`/`mapErrorToStatusCode(error)` pattern across every function in this audit. `createHttpError(message, status)` (from `_shared/org.ts`) is used inconsistently — some functions (`mock-publish`, `ai-brand-consistency-check`) use it for most expected failure modes (giving precise status codes), while others (`generate-post-metadata`) throw plain `Error`s for access-denial cases, relying on whatever generic mapping `mapErrorToStatusCode` applies to an untyped error (likely a blanket 500 rather than 403/404 — UNCLEAR without reading that helper directly).
- **Media-specific normalization**: `media.service.js`'s `normalizeMediaErrorMessage` intercepts fal.ai/provider-quota and missing-API-key error text patterns and rewrites them into specific, actionable messages before they ever reach a toast — the only client-side error-message rewriting layer found in this audit; no equivalent exists for LLM-provider errors (a Groq/Claude outage surfaces its raw error text to the user via toast).

## Logging / analytics

- No dedicated analytics-event system (e.g., Segment/PostHog/Amplitude calls) was found anywhere in the reviewed Generate/Studio files. The closest analog is the `socialai:data-sync` window `CustomEvent`, dispatched with a `reason` string (`'generation-completed'`, `'draft-saved'`, `'post-published'`, etc.) after nearly every mutating action — but this is purely an in-app cache-invalidation signal (consumed by `GeneratePageV2`'s search-index refresh and presumably other pages' own data-refresh listeners), not a telemetry pipeline; nothing is sent to any external analytics service from this signal.
- Server-side, every edge function `console.error`s on failure with a `[function-name] error` prefix — these presumably surface in Supabase's function logs, but no structured logging/tracing (request ids, span correlation) was observed beyond that.
- `posts_audit`/`generations_audit` DB triggers write to an `audit_logs` table on insert/update — this is the closest thing to a durable event trail for this domain, but it's DB-level, not application/analytics-level, and wasn't inspected further (out of scope).

## Credit/quota/billing checks

- **Authoritative check is always server-side**, in the edge function, against `user_credits.balance` (personal) or the org credit pool via `ensureCreditsAvailable`/`recordCreditUsage` (org, `ai-brand-consistency-check` only, among this page's functions) — never the client's own `estimateGenerationCost` computation, which Stage 2/3a already flagged as explicitly non-authoritative and fails open (`canAfford = true`) while the real balance hasn't loaded yet.
- Deduction happens via the `deduct_credits` RPC inside `generateImage`/`editImage`/`generateVideo`, categorized (`image|carousel|video|edit`) for the spend-breakdown UI elsewhere in the app. Notably, **generating a content plan, enhancing a prompt, generating post metadata/captions, scoring/optimizing SEO** are all **free** (no credit check or deduction) for personal-workspace users in this flow — only the actual media-rendering calls cost credits. The one exception, `ai-brand-consistency-check`, costs org credits specifically because it's gated behind `ensureCreditsAvailable`.
- No visible **rate limiting** (per-user request throttling) in any edge function reviewed — repeated rapid calls to any of these functions are limited only by whatever platform-level (Supabase/Deno Deploy) defaults exist, not by application code.

## Feature flags

- None found anywhere in the reviewed Generate/Studio code paths — no `useFeatureFlag`/LaunchDarkly/GrowthBook-style gating on any of this page's behavior. Mode availability (image/carousel/video/edit/image-to-video) is a static list (`CONTENT_TYPES` in `mediaGenerationOptions.js`), not flag-controlled.

## TODOs, hacks, and commented-out code worth flagging

- **`slideSelection` state** in `StudioPage.jsx` — declared, never read or written elsewhere in the reviewed file; likely dead state left over from an incomplete feature.
- **`postPanelOpen` state** in `GeneratePageV2.jsx` — set in two places, never read by any rendering logic found; the actual publish-panel visibility is driven by `StudioPage.jsx`'s own independent `studioStage` variable. Likely dead/vestigial state from before the design-system-v2 rebuild.
- **Disconnected event contracts**: `socialai:seed-prompt` and `socialai:activate-generation-edit` are dispatched by `GeneratePageV2.jsx`'s route-state handoff effect but have no listener anywhere in the reviewed `StudioPage.jsx`/other Studio files — these look like surviving pieces of an event-based integration contract from the pre-rebuild implementation (`BrandosseGenerateStudio.jsx`, referenced only in a comment as already deleted) that was never re-wired into the new `ui-v2` Studio.
- **Video job "background survival" claim vs. implementation**: as detailed in Stage 3a, the UI copy in two places (header indicator, Video Jobs drawer) asserts that video generation "keeps processing in the background" / "even if you leave Studio," but the actual implementation is a single synchronous edge-function await with no server-side job ID or client-side polling — the claim happens to mostly hold in practice (because the promise keeps running as long as the tab/store survive) but is not backed by the deliberate async-job architecture the copy implies. Related: `videoJobState.pollInterval` and the realtime subscription's video-specific completion/failure branch appear to be **unused scaffolding from a prior polling-based design** that the current synchronous edge function superseded without full cleanup.
- **Schedule dialog writes `scheduled_at: null`**: as detailed in Stage 3b, the dedicated "Schedule this post" modal collects a date/time but the underlying `saveDraft()` call it invokes explicitly hardcodes `scheduled_at: null` in its DB payload — the collected date is applied to in-memory `postProduction.scheduleDate` but never persisted through this particular code path. This reads as a genuine functional bug, not an intentional simplification, since a *second*, different code path (`publishContent`, when `postProduction.scheduleDate` is already set) does correctly persist `scheduled_at`.
- **"Regenerate whole carousel" button** (`StudioPage.jsx`) calls `handleModeChange('carousel')`, which only resets mode/settings state — it does not actually trigger a new generation despite its label implying it would.
- **Two independently-computed "the same" SEO score** for optimized content: `optimize-seo`'s own inline self-score and the separate `seo-score` call the client chains immediately afterward compute overall score via different normalization logic (weighted-breakdown recompute vs. simpler top-level-field correction) — flagged in Stage 5c as a real double-LLM-call inefficiency and a potential source of visibly inconsistent scores for the same content shown moments apart in the UI.
- **Client/DB duplication candidates**: `ensureDraftForGeneration` (client) vs. `ensure_draft_post_for_generation()` (DB trigger on `generations`); `ensureLibraryRowsForPosts` (client) vs. `create_library_item_from_post()` (DB trigger on `posts`). Both pairs appear to independently guarantee the same outcome — worth a follow-up investigation into whether the client-side calls are still load-bearing (e.g., for the fire-and-forget metadata-generation side effect bundled into `ensureDraftForGeneration`, which the DB trigger obviously cannot perform) or purely redundant.
- **Vestigial browser-side LLM code**: `groqClient.js`'s `callWithFailover`/`callProvider`/`enhancePromptWithBrand`/`callGroqVisionJSON` paths are permanently unreachable (`BROWSER_GROQ_TOKEN = ''`, hardcoded empty) and not invoked by any traced flow in this page — dead code retained after migrating LLM calls server-side, per its own header comment.
