# Stage 4 — API Layer

## Approach

No React Query/SWR usage in this page's own code — all data access is direct `supabase.from(...)` calls (Postgres via PostgREST) or `supabase.functions.invoke(...)` (Edge Functions), executed inside Zustand store actions or component effects. TanStack Query is configured globally in `NextAppProviders.jsx` but bypassed here. There is no request caching layer beyond the Zustand store itself holding the last-fetched arrays in memory, and `edgeFunctionClient.js`'s sessionStorage-based "unavailable" cooldown (5 min) which normalizes errors but is not observed being actively armed by this page's call sites (`markEdgeFunctionUnavailable` exists but no call to it was found in `SessionStore.js`).

### Non-edge-function server call
- `sessionTitleService.js: generateSessionTitle` calls `POST /api/session-title` — a **Next.js API route**, not a Supabase Edge Function — used only when creating a brand-new session from the first prompt (`ensureSessionFromPromptInput`). Falls back to `'Untitled Session'` on any non-OK response or thrown error (never surfaces an error to the user).
- `groqClient.js` also contains a `callWithFailover`/`callProvider` path that would call `https://api.groq.com/...` directly from the browser — but `GROQ_TOKEN` is hardcoded to an empty string (`BROWSER_GROQ_TOKEN = ''`) with a comment stating browser-exposed provider keys are intentionally disabled; `providerOrder()` returns an empty array in this state, so `callWithFailover` always throws `'No browser LLM provider is configured...'` if actually invoked. The functions that route through this (`enhancePromptWithBrand`, `callGroqJSON`, `callGroqVisionJSON`) are **not called anywhere in this page's traced flow** — only `callGroqContentPlan`/`callGroqRevision` are used, and those go through the edge-function path (`invokeContentPlanFunction`), not `callWithFailover`. This looks like dead/vestigial browser-LLM code retained after migrating LLM calls server-side.
- `ApiService.js: enhancePrompt` (imported as `apiEnhancePrompt`) — legacy fallback only reached if `enhance-prompt` edge function itself errors inside `SessionStore.js`'s `enhancePrompt` action; its own internals were not read in this audit (out of scope, only its role as a fallback is confirmed via the import/call site).

## Caching, retries, polling, streaming

- **Caching**: none beyond the Zustand store's in-memory arrays; `fetchGenerations` has a `silent` option to skip flipping the loading flag when data already exists for the same session (a cheap "don't flash a spinner on a refetch" optimization, not a cache-hit/cache-miss mechanism).
- **Retries**: none automatic anywhere in this flow — every edge-function/Postgres call is a single attempt; all "retry" behavior is user-initiated (re-clicking Generate/Retry buttons), not built into the client or a retry wrapper.
- **Polling**: `useCreditBalance`/its sibling category hook fetch once on mount and thereafter rely purely on realtime `postgres_changes` push events — no interval polling exists for credits, generations, or video jobs (see Stage 3a's note that `videoJobState.pollInterval` is legacy scaffolding that's never actually populated with a real interval in the current implementation).
- **Streaming**: none — no SSE/streaming responses; `generateImage`/`generateVideo`/`generate-content-plan` are all single request/response Edge Function invocations (per `media.service.js`'s own comment, the video edge function polls fal.ai's queue **server-side**, synchronously, before responding).

## Realtime

- Single Supabase Realtime channel (`generations_updates`) via `subscribeToGenerations` (`SessionStore.js:2749-2834`), subscribed to Postgres `postgres_changes` events (`event: '*'`) on the `generations` table (no server-side filter — the client receives changes to **all** rows in the table, filtering by `updated.session_id === activeSession?.id` and `updated.id === videoJobState.generationId` client-side after the fact).
- Subscribed once from `GeneratePageV2.jsx`'s mount effect (`useEffect(() => { const unsubscribe = subscribeToGenerations(); return unsubscribe; }, [subscribeToGenerations])`) — one channel per page mount, torn down (`supabase.removeChannel`) on unmount.
- On `INSERT`/`UPDATE`: merges the row into `activeGenerations` (matching by id, replacing or appending), updates `selectedGeneration` if it matches the currently-selected id, and — if the row just transitioned to `COMPLETED` with a `storage_path` — fires `ensureDraftForGeneration` in the background (covers the case where a generation completes via a path other than the direct client-awaited call, e.g., if the pipeline's server-side write lands while the client's own await already moved on, or a second browser tab/session triggered it).
- Separately handles the video-specific completion/failure branches tied to `videoJobState.generationId` (see Stage 3a's note that this branch is largely dead code given the current synchronous video edge function design, since `pollInterval` is never set to a live interval and the direct-await success/failure path already updates `videoJobState` before this handler would ever see the corresponding realtime event).
- **No filter by `organization_id`/workspace scope on the realtime subscription itself** — a personal-workspace session would still receive realtime payloads for org-scoped generations table changes (globally, for any row in the table across all users/orgs), relying entirely on the client-side `session_id === activeSession?.id` check to discard irrelevant rows. This means every connected client receives every row change in the table verbatim (before filtering) — a potential data-exposure surface worth flagging to a security reviewer (out of scope for this architecture-only stage, noted here since it's directly observable from this page's own subscription code).
