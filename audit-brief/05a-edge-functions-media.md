# Stage 5a ‚Äî Edge Functions: Media Generation (fal.ai)

All three functions share the same skeleton: CORS preflight ‚Üí require POST ‚Üí auth via bearer token ‚Üí admin (service-role) Supabase client ‚Üí credit check ‚Üí optional Claude-Haiku prompt enhancement (`callPromptEngine`, `_shared/llm.ts`) ‚Üí fal.ai call (`_shared/fal.service.ts`) ‚Üí fetch the provider's result URL and re-upload to Supabase Storage bucket `generated_assets` ‚Üí optionally insert a `generations` row ‚Üí deduct credits via `deduct_credits` RPC ‚Üí return a response with several duplicate key aliases (both camelCase and snake_case) for backward compatibility with different client callers.

**Important cross-cutting fact for the whole file**: `src/services/media.service.js` (the client wrapper this page uses) always sends `record_generation: false` in every call to `generateImage`/`editImage`/`generateVideo`. This means **the edge functions' own `generations` insert branch never runs for traffic coming from this page** ‚Äî that insert path exists for some other caller (unconfirmed, out of scope) that wants the edge function itself to own the DB row. For Studio's flows, the actual `generations` row is instead created by the **calling code** (`generationPipeline.js` for image/carousel, or directly in `SessionStore.js` for edit/video) ‚Äî see Stage 3a. This is worth flagging: the edge function's credit-deduction and generation-recording logic are decoupled per-caller-choice, and the two paths use different columns in places (`output_url` vs `storage_path` handling, `provider`/`provider_model` naming) ‚Äî a discrepancy an architecture reviewer should verify doesn't cause any generation to be billed without a corresponding DB row, or vice versa.

## `generateImage` (`supabase/functions/generateImage/index.ts`)

- **Trigger**: `POST`, invoked by `media.service.js: generateImages()`, once per requested image/slide.
- **Auth**: `requireUser(authClient)` ‚Äî rejects unauthenticated callers (`_shared/supabase.ts`, not read in full ‚Äî behavior inferred from usage).
- **Credit check**: reads `user_credits.balance` for the user via the **admin** client (bypasses RLS, so this is the authoritative check, not the client's own estimate); returns HTTP 402 `{ error: "Insufficient credits" }` if `balance < CREDITS_PER_IMAGE` (1).
- **Prompt enhancement**: unless `enhance_prompt === false` (client never sets this, so it always runs), calls `callPromptEngine` (Claude Haiku per file header comment) to rewrite the prompt with brand context injected; on any failure, silently falls back to the raw prompt (`catch (_) { finalPrompt = rawPrompt; }` ‚Äî non-critical, no error surfaced).
- **Generation**: `generateImageByModel(imageModel, {...})` ‚Äî model selectable via `image_model` (client sends `settings.imageModel || 'ideogram'`); supports Recraft-specific `brand_colors` extraction from the brand kit's hex color fields (`brand_colors`/`color_palette`/`colors`/`palette`/`primary_color`/`secondary_color`/`accent_color`), converted to RGB triples, capped at 5.
- **Logo compositing**: if `logo_url` is provided (not sent by this page's client code ‚Äî `media.service.js` never passes it), composites a brand logo onto the generated image via `compositeLogo` (`_shared/composite.ts`); failure here is caught and logged as a warning, falling back to the uncomposited base image ‚Äî never blocks the response.
- **Storage**: fetches the fal.ai result URL server-side, uploads raw bytes to `generated_assets/{userId}/{timestamp}_{model}.{ext}` with `upsert: true`, retrieves the public URL.
- **DB writes**: (only when `record_generation !== false`, i.e., not from this page) inserts into `generations` with `provider`, `provider_model`, `enhanced_prompt` (only set if enhancement actually changed the prompt), `metadata.cost_usd`. Always calls `deduct_credits` RPC regardless of `record_generation`, tagging the ledger category as `"carousel"` if `body.category === "carousel"` else `"image"` ‚Äî this is how the credit ledger's category breakdown (seen in `useCreditSpendByCategory`) gets populated.
- **Response**: returns the public URL under many duplicate keys (`url`, `publicUrl`, `public_url`) plus `storagePath`/`storage_path`, provider/model info under duplicate key names, `credits_used`, `credits_remaining` (computed as `currentCredits - CREDITS_PER_IMAGE` ‚Äî **not re-read from DB after the RPC**, so if the RPC's actual deduction differs from the assumed amount for any reason, this reported balance could be inaccurs, though in practice both are hardcoded to the same constant).
- **Error handling**: single top-level `try/catch`; any thrown error (missing prompt, insufficient credits already returned early, insufuent image URL, storage upload failure, DB insert failure) is normalized via `toErrorPayload`/`mapErrorToStatusCode` (`_shared/http.ts`, not read ‚Äî behavior inferred) and returned as a JSON error response with an appropriate status code; also `console.error`'d server-side.
- **Rate limiting/timeouts**: none visible in this file ‚Äî no explicit timeout wrapper around the fal.ai call or the storage upload; relies on whatever default timeouts fal.ai's SDK/Deno's `fetch` impose.

## `editImage` (`supabase/functions/editImage/index.ts`)

- Same skeleton as `generateImage`, adapted for FLUX.1 Kontext Pro image editing.
- **Credit cost**: `CREDITS_PER_EDIT = 3` (matches the client-side estimate in `mediaGenerationOptions.js`).
- **Required fields**: `prompt` and `sourceImageUrl` are both hard-required (400 if either missing) ‚Äî mirrors the client-side `validatePreflight` check, but is the actual authoritative validation.
- **Prompt enhancement**: same Claude Haiku pattern, tuned instructions for edit precision ("Reference brand visual style only where it doesn't conflict with the requested edit"), capped at 120 words / 200 max tokens.
- **Generation**: `generateImageEdit({ prompt, image_url: sourceImageUrl, aspect_ratio })` ‚Äî no model choice param (single fixed model, `FAL_MODELS.imageEditKontext`).
- **Storage/DB/credits**: identical pattern to `generateImage`; `metadata.source_image_url` recorded when the (unused-by-this-page) DB-insert branch runs.
- **Response**: `publicUrl`, `storagePath`, provider info, `credits_used: 3`, `credits_remaining`.

## `generateVideo` (`supabase/functions/generateVideo/index.ts`)

- Same skeleton, most complex of the three due to the standard/premium tier-substitution logic.
- **Tier resolution** (`:66-77`): `requestedQuality` from `body.quality` (default `'standard'`); `isI2V = Boolean(body.image_url)`. Hailuo 2.3 (the "standard" engine) is **image-to-video only** ‚Äî a standard request with no image is silently (from the requester's perspective, though clearly documented server-side) upgraded to `quality: 'premium'` (Kling 2.5 Pro) and billed at the premium rate (`tierUpgraded = true`). This exactly matches the client-side comment/UI copy in `StudioPage.jsx` ("Standard requires a source image... renders at premium quality and is billed accordingly").
- **Credit check**: checked against **the tier that will actually render** (post-upgrade), not the one requested ‚Äî `creditsNeeded` is `CREDITS_PRO_VIDEO` (15) or `CREDITS_STD_VIDEO` (5) accordingly. Insufficient-credit error message is tier-upgrade-aware, explaining *why* more credits are needed than the user might expect if they only requested "standard."
- **Prompt enhancement**: Claude Haiku, instructions vary by target engine (Kling vs Hailuo) and by whether it's image-to-video (adds "the video starts from a reference image" framing).
- **Generation**: three possible fal.ai calls depending on `quality`/`isI2V` ‚Äî `generateVideoKlingI2V`, `generateVideoKling`, or `generateVideoHailuo`. Cost computed either per-second (Kling, `FAL_COST_USD.videoKlingPerSec * duration`) or per-clip (Hailuo, flat `FAL_COST_USD.videoHailouPerClip`).
- **Storage**: fetches result as a `.blob()` (vs. `.arrayBuffer()` in the image functions ‚Äî inconsistent byte-handling style across the three functions, though functionally equivalent for the Supabase Storage client), uploads as `video/mp4`.
- **DB writes**: (unused-by-this-page branch) records `quality`, `requested_quality`, `tier_upgraded`, `duration`, `is_image_to_video`, `cost_usd` in `metadata`. Credit deduction always runs, tagged category `"video"`.
- **Response**: extensive duplicate-key aliasing again, plus `tier_upgraded`/`tier_upgrade_reason` (human-readable explanation string) and `requested_quality` vs `quality` (actual) ‚Äî this is exactly what `media.service.js: createVideoJob` reads to decide whether to show the client-side "rendered at premium instead" toast.
- **Error handling/rate limiting/timeouts**: same pattern as the other two ‚Äî single try/catch, no explicit rate limiting or timeout wrapper visible in this file.

## Shared dependency notes (not fully read ‚Äî out of Stage 5's per-function scope but referenced by all three)

- `supabase/functions/_shared/fal.service.ts` ‚Äî the actual fal.ai SDK integration (model registry, cost table `FAL_COST_USD`, aspect-ratio mapping `aspectToFalImageSize`) ‚Äî this is the single source of truth for supported models/params that `src/config/mediaGenerationOptions.js`'s header comment explicitly tells frontend maintainers to consult before changing UI options.
- `supabase/functions/_shared/llm.ts` (`callPromptEngine`) ‚Äî the Claude Haiku prompt-enhancement helper shared by all three media functions; failures here are always treated as non-critical (silent fallback to the raw/unenhanced prompt), so a Claude outage degrades prompt quality but never blocks generation.
- `supabase/functions/_shared/composite.ts` (`compositeLogo`) ‚Äî logo-overlay utility used only by `generateImage`, not exercised by this page's current UI (no logo_url is ever sent).
- `supabase/functions/_shared/supabase.ts` (`createAdminClient`, `createAuthClient`, `requireUser`) ‚Äî shared auth/client-construction helpers.
- `supabase/functions/_shared/http.ts` (`handleCors`, `jsonResponse`, `mapErrorToStatusCode`, `parseJsonBody`, `toErrorPayload`) ‚Äî shared HTTP plumbing.
- `deduct_credits` ‚Äî a Postgres RPC function (not a table read in this stage) ‚Äî the actual atomic credit-deduction mechanism; its internals (row locking, insufficient-balance re-check, `credit_transactions` insert) are not traced here (would require reading the migration/function SQL, out of scope for this edge-function-focused stage ‚Äî see Stage 6 for the `credit_transactions` table-level note).
