# Fix Log

## Fix 1 — Schedule dialog silently discards the scheduled date

**Reality check vs. audit**: The audit assumed "if NONE exists, do not build" a downstream scheduler. Reality differs — a real scheduler already exists: `process_scheduled_posts()` (cron, every minute, `supabase/migrations/20260710140000_create_process_scheduled_posts.sql`) selects `posts WHERE status='scheduled' AND scheduled_at <= now()`, flips them to `publishing`, and calls `dispatch_scheduled_post()` → `publish-post` edge function (which itself routes to either the mock-publish flow or a real platform API depending on `connected_accounts.is_mock`). This means the bug was worse than "user sees a wrong toast" — it silently meant **no post scheduled through Studio was ever actually published**, because the cron's query could never match a row whose `scheduled_at` was always `NULL`.

**What changed**:
- Added `resolveScheduledPostFields(postProduction, { immediateStatus, immediateScheduledAt })` — a single helper that decides `{ scheduled_at, status }` from `postProduction.scheduleDate`. This is now the **one and only** place that builds the scheduled-vs-not payload.
- `saveDraft()` now calls this helper (`immediateStatus: DRAFT, immediateScheduledAt: null`) instead of hardcoding `scheduled_at: null` — when `postProduction.scheduleDate` is set, it writes `scheduled_at` + `status: 'scheduled'`; otherwise unchanged (`scheduled_at: null`, `status: 'draft'`). Return value now reports `status: 'scheduled'`/message `'Scheduled successfully!'` when the scheduled branch ran, vs. the original `'Saved to drafts!'`/`status: 'draft'` otherwise. `dispatchContentSync` fires `'post-scheduled'` instead of `'draft-saved'` on the scheduled branch.
- `publishContent()` now calls the **same** helper (`immediateStatus: PUBLISHING`, default `immediateScheduledAt: now()`) instead of its own inline `scheduleDate`/`status` computation — removes the second divergent payload builder.
- `StudioPage.jsx: handleConfirmSchedule` no longer prefers the generic store message; it now checks `result?.status === 'scheduled'` and shows `"Scheduled for {date} {time}"` (using the exact values the user picked in the dialog), falling back to the store's own message only on the (should-never-happen) non-scheduled branch.

**Status machine / DB**: Verified `postStatusMachine.js` already allows `DRAFT → SCHEDULED` and `FAILED → SCHEDULED` (`TRANSITIONS[DRAFT]`/`TRANSITIONS[FAILED]` both include `SCHEDULED`) — no machine change needed. Verified `lock_terminal_posts()` (the `enforce_post_lifecycle` trigger body, `20260227090000_calendar_library_alignment.sql:221-237`) only blocks transitions *away from* `published`/`publishing` — does not touch draft→scheduled. Verified `idx_posts_unique_draft_per_generation_account` is a **partial** unique index (`WHERE status = 'draft'`) — promoting a draft to `scheduled` removes the row from that index's scope, so no conflict. **No migration was needed for this fix.**

**Layers touched**: FE (`StudioPage.jsx`), Store (`SessionStore.js`). No DB migration (verified unnecessary, see above). No edge function changes.

**Journey verified (traced, not executed in a live environment)**: user generates → continues to post production → Schedule → picks date/time → Confirm → `updatePostProduction({ scheduleDate })` (synchronous Zustand set) → `saveDraft()` reads the now-current `postProduction.scheduleDate` → resolves to `{ scheduled_at, status: 'scheduled' }` → upsert writes both columns → toast "Scheduled for {date} {time}" → stage returns to 'brief'. Reload + repurpose/edit handoff on that post: `hydratePostProductionFromGeneration` already reads `preferred.status === SCHEDULED ? preferred.scheduled_at : ...` (pre-existing code, now fed real data) → the scheduled date now round-trips correctly.

**Known corner case (not a bug, flagging for visibility)**: `postProduction.scheduleDate` can also be pre-populated by the Calendar "new post on this date" handoff (`prefillScheduleDate` in `GeneratePageV2.jsx`) *before* the user ever opens the Schedule dialog. Because `saveDraft` is now schedule-aware, clicking the plain "Save as draft" footer button after arriving via that handoff will now schedule the post for the prefilled date rather than saving a plain draft. This matches the literal fix spec (schedule-aware `saveDraft`) and arguably matches the handoff's intent (the user picked a calendar slot), but it is a real behavior change or a plain "Save as draft" click. No action taken — out of this fix's stated scope; flagging for product-decision follow-up.

**Downstream scheduler**: EXISTS (see reality-check note above) — `process_scheduled_posts` cron → `dispatch_scheduled_post` → `publish-post` edge function. This fix is what makes that existing pipeline actually reachable from Studio-created scheduled posts; previously it was silently dead for this entry point.

## Fix 2 — enhance-prompt edge function has no authentication

**Reality check vs. audit**: Confirmed still accurate — `enhance-prompt/index.ts` was the only edge function in this flow with no `createAuthClient`/`requireUser` call. Confirmed the exact auth pattern used by every sibling function (`createAuthClient(req.headers.get("Authorization"))` then `await requireUser(authClient)`, both from `_shared/supabase.ts`) and confirmed `mapErrorToStatusCode` (`_shared/http.ts`) maps any error whose message contains `"unauthorized"` (case-insensitive) to HTTP 401 — both `createAuthClient`'s missing-header error and `requireUser`'s failure message contain that substring, so this function now fails exactly the same way (401, same body shape via `toErrorPayload`) as every sibling.

**What changed**:
- `supabase/functions/enhance-prompt/index.ts`: added `import { createAuthClient, requireUser } from "../_shared/supabase.ts";` and, first thing inside the try block (before parsing the body), `const authClient = createAuthClient(...); await requireUser(authClient);`. No other logic in the function touched.
- `src/services/edgeFunctionClient.js`: exported the previously-private `getEdgeStatus(error)` helper (was already used internally by `isEdgeFunctionUnavailable`/`normalizeEdgeFunctionError`) so the store could reuse it rather than re-implementing status extraction.
- `src/stores/SessionStore.js: enhancePrompt`: after the `supabase.functions.invoke('enhance-prompt', ...)` call, added an explicit check — if the response errored with status `401`/`403`, the action now shows `toast.error('Your session has expired. Please log in again to enhance prompts.')` and returns the original prompt unchanged, **without** calling `apiEnhancePrompt` (the legacy fallback). Any other error (network failure, function unreachable, 5xx) still falls through to the pre-existing fallback chain exactly as before.

**Client already sent the auth token correctly** — verified `src/services/supabaseClient.js`'s custom `global.fetch` override only intercepts requests to `/auth/v1/*` (a timeout/retry wrapper for the auth service itself); it passes `functions.invoke` requests straight through to the real `fetch`, and supabase-js's `functions.invoke()` attaches the current session's `Authorization: Bearer <access_token>` header itself before that call — no header stripping anywhere in this codebase. So no frontend change was needed to "start sending" a token; it always did.

**`ApiService.js: enhancePrompt` internals, now read and documented (was never traced before)**: it is effectively inert, not a security risk. `API_KEYS.groq` is hardcoded to `''`; `generateTextWithGroq` checks `if (!API_KEYS.groq)` first and, when empty, logs a console warning and returns a **canned local string** (`mockTextResponse()`) — it never makes a network call to any endpoint, authenticated or not. So the legacy fallback path does not "call an unauthenticated legacy endpoint" as the audit worried it might; it silently returns a placeholder sentence ("This is a mock AI response...") as the "enhanced" prompt whenever it's reached (i.e. whenever `enhance-prompt` itself is unreachable). No security follow-up needed here, but this is worth flagging as its own (separate, out-of-scope) UX bug: a genuine availability failure of `enhance-prompt` currently replaces the user's real prompt with an obviously-fake placeholder string instead of leaving the original prompt untouched.

**Also discovered, NOT fixed (out of scope for Fix 2)**: `StudioPage.jsx: handleEnhance` reads `result?.enhanced` (`StudioPage.jsx:208`), but the store's `enhancePrompt` action returns `{ enhancedPrompt, suggestions }` — there is no `enhanced` key, so `handleEnhance` never actually applies the enhanced prompt to the textarea on success today, regardless of this fix. This is a pre-existing, independent bug in the success path, not touched here since it's unrelated to authentication — flagging for a future fix.

**Layers touched**: BE (`enhance-prompt/index.ts`), FE/Store (`SessionStore.js`, `edgeFunctionClient.js`). No DB changes.

**Journey verified (traced)**:
- Logged-in user clicks "Enhance prompt" → `supabase.functions.invoke` attaches the real session token → edge function's `requireUser` succeeds → unchanged behavior from the user's perspective.
- Anonymous/unauthenticated call (e.g. curl with no `Authorization` header, or an invalid one) → `createAuthClient` throws `"Unauthorized: missing Authorization header"` before any `parseJsonBody`/LLM call runs → `toErrorPayload`/`mapErrorToStatusCode` → 401 response, zero LLM tokens spent.
- Expired-session user clicks "Enhance prompt" → edge function returns 401 → store detects `getEdgeStatus(error) === 401` → shows "Your session has expired..." toast, returns the original prompt unchanged → does **not** call `apiEnhancePrompt`.

**RATE LIMITING CHECKPOINT**: None needed for this fix — adding `requireUser` does not require rate limiting to be "correct" (per-user request throttling is an orthogonal concern to authentication). No checkpoint triggered.

Removing the dispatches from `GeneratePageV2.jsx` does not affect `OrgGenerateComposer.jsx`/`GenerationCanvas.jsx` at all, since that pair dispatches and listens to its own independent copies of the same event names. Left `OrgGenerateComposer.jsx` and `GenerationCanvas.jsx` completely untouched (out of scope — different page/flow).

**Layers touched**: FE (`GeneratePageV2.jsx`, `StudioPage.jsx`), Store (`SessionStore.js`). No BE/DB changes — this fix is purely about the client-side handoff wiring.

**Reality check vs. audit**: Confirmed exactly as described — `StudioPage.jsx:717` (current line) called `onClick={() => handleModeChange("carousel")}`, which only resets `settings`/mode state and never calls any `start*Generation` action.

## Fix 5 — "Save as draft" (brief panel) throws raw 'No generation selected'

**Reality check vs. audit**: Confirmed exactly as described — the brief panel's "Save as draft without generating" button (`StudioPage.jsx`, brief panel footer) called `saveDraft()` unconditionally with no `disabled` guard, and `saveDraft()` (`SessionStore.js`) throws `new Error('No generation selected')` when `!selectedGeneration`, which would surface verbatim via `toast.error(err?.message || "Could not save draft")`.

**Decision (a) vs (b), with reasoning**: Checked whether a promptless-draft mechanism already exists via `sessions.metadata.draft_prompt` before deciding. It exists, but only as a **write-once field set at session creation time** (`ensureSessionFromPromptInput`/`createNewSession` in `SessionStore.js` stamp `metadata: { draft_prompt: prompt, title_source: ... }` the moment a *new* session is created from a first prompt). There is no code path anywhere that (1) updates `draft_prompt` again later when the user edits the prompt textarea, or (2) reads `draft_prompt` back to restore/display it. So this is not "an existing promptless-draft system that's simply unwired to this button" — it's a thin, one-time session-naming aid that does not do what this button's label implies. Building real read/write wiring for a general "save my current prompt for later" feature would itself be a new promptless-draft system, which is explicitly out of scope for this pass. **Went with option (a)**: disable the button when no generation is selected.

**What changed**:
- `src/pages/Studio/StudioPage.jsx`: added `disabled={!selectedGeneration}` and `title={selectedGeneration ? undefined : "Generate first, then save as a draft"}` (native tooltip — no dedicated Tooltip component exists in `ui-v2`'s exported set, so this uses the plain HTML `title` attribute rather than introducing a new UI pattern) to the "Save as draft without generating" button. When a generation *is* selected, the button is enabled and behaves exactly as before (calls `saveDraft()`, success/error toasts unchanged).
- `src/stores/SessionStore.js`: `saveDraft()`'s guard message changed from `'No generation selected'` to `'Select a generation to save as a draft.'` — a user-appropriate message in case any other, not-yet-audited call site ever reaches this guard without going through the now-disabled button. Verified (via a codebase-wide search) that no other code branches on the exact string `'No generation selected'`, so this rename is safe.

**Future item logged (not built)**: a real "save my typed prompt/brief as a draft before generating" feature — would need `postProduction`/`sessions.metadata` (or a new lightweight table) to actually persist and later restore free-typed prompt text independent of any generation. Flagging for a future, dedicated pass; not attempted here per the fix's explicit scope boundary.

**Layers touched**: FE (`StudioPage.jsx`), Store (`SessionStore.js`, message text only). No BE/DB changes.

**Journey verified (traced)**: fresh page, no generation selected → button is disabled with a hover tooltip explaining why → no click possible, no raw error can reach the user. After generating and selecting a generation → button enables → click → `saveDraft()` succeeds exactly as before Fix 5 (and, per Fix 1, is now schedule-aware if `postProduction.scheduleDate` happens to be set, consistent with every other `saveDraft()` call site).

## Fix 6 — Drawer "resume session" doesn't update the URL

**Reality check vs. audit**: Confirmed exactly as described — `SessionHistoryDrawer`'s `onResume` prop (wired in `StudioPage.jsx`) called `loadSession(s.id)` directly and only closed the drawer, never navigating. Also confirmed the two other session-mutating call sites the fix asked to sweep: `onNewSession` called `createNewSession(...)` with no navigation, and the delete-session confirm modal's button called `deleteSession(...)` with no navigation. Re-verified `GeneratePageV2.jsx`'s URL-driven init effect (`useEffect(..., [user?.id, sessionId, loadSession, createNewSession, clearActiveSession, navigate, location.state])`) already lists `sessionId` as a dependency and already handles every case relevant here: a truthy `sessionId` triggers `loadSession(sessionId)`; if that returns `null` and no route state requires a persisted session, it calls `clearActiveSession()` and navigates to bare `/app/generate`. No changes were needed to that effect itself — it was already correctly URL-reactive; only the call sites that mutate the active session without going through the URL needed fixing.

**What changed** (`src/pages/Studio/StudioPage.jsx`, all three within `SessionHistoryDrawer`'s props / the delete-session modal):
- `onResume`: now calls `navigate(`/app/generate/${s.id}`)` and closes the drawer — no longer calls `loadSession` directly. The existing URL-driven init effect in `GeneratePageV2.jsx` performs the actual load once `sessionId` changes. Removed the now-unused `loadSession` from `StudioBody`'s store destructuring (it was only ever called from this one call site).
- `onNewSession`: still calls `createNewSession(...)` (still needed — a session has to actually exist before you can navigate to its URL), but now captures the returned session and calls `navigate(`/app/generate/${created.id}`)` afterward so the URL reflects the new session immediately.
- Delete-session confirm button: captures `wasActive = activeSession?.id === deleteSessionTarget.id` *before* calling `deleteSession(...)`, and if the deleted session was the active one, navigates to bare `/app/generate` afterward. Deleting a session that is **not** the active one (e.g. some older session in the drawer while a different one is open) correctly leaves the URL untouched, since the active session/URL relationship wasn't affected.
- Did not touch the URL-hash (`#<generationId>`) mechanics anywhere in this fix, per the explicit scope boundary.

**Verified the init effect does not double-load/flash incorrect content** for either navigation-driven path (traced, not run live):
- Resume: `navigate` changes `sessionId` → effect re-runs → `loadSession(s.id)` finds the session already cached in the store's `sessions` array (populated by the drawer's own `fetchSessions`) — no redundant session-row fetch; `isSameSession` is false (switching away from whatever was previously active) so `activeGenerations`/`selectedGeneration` are correctly reset before the new session's generations are fetched (not silently, since there's nothing cached yet for the new session) — no flash of the old session's content.
- New session: `navigate` to the just-created id → effect re-runs → `loadSession` finds `isSameSession === true` (the store's `activeSession` was already set by `createNewSession`) → skips resetting `activeGenerations` (already empty) → performs one harmless confirmatory fetch (returns empty, matching reality) — no incorrect flash, just one small redundant round-trip.

**Layers touched**: FE only (`StudioPage.jsx`). No BE/DB changes — this fix is entirely about which client-side code path is allowed to change the active session.

**Journey verified (traced)**: open drawer → resume an older session → URL becomes `/app/generate/<thatId>` → that session's generations load → hard refresh → same session loads (URL is now the persisted source of truth, store rehydrates from it on a fresh module load). Create new session from drawer → URL updates to the new session's id → refresh persists it. Delete the active session → URL returns to bare `/app/generate` → refresh shows the clean/no-session state (via `clearActiveSession`'s existing behavior, unchanged). Delete a non-active session from the drawer while a different session is open → URL and active session remain on the one that was already open, unaffected.

**Could not verify end-to-end in a running browser** (no dev server/browser session available in this pass) — verified by static trace of the effect's dependency array, the store actions' return values, and the navigation call shapes only. Flagging for a manual QA pass alongside Fix 3's.

## Fix 2 — Typed HTTP errors in edge functions (CORRECTNESS/OBSERVABILITY)

**Reality check vs. audit**: Confirmed accurate, and fully read `_shared/http.ts`/`_shared/org.ts` this time (the audit itself flagged that it never had). Findings documented below exactly as required.

### What `mapErrorToStatusCode` actually does with an untyped `Error` (read in full, not inferred)

`mapErrorToStatusCode(error)` first checks `"statusCode" in error` — if present (i.e. the error came from `createHttpError`), that value wins outright, no further logic runs. Only if absent does it fall back to lower-casing `error.message` and substring-matching: `"unauthorized"` → 401, `"forbidden"` → 403, `"not found"` → 404, any of `"bad request"`/`"invalid"`/`"missing"` → 400, `"rate limit"`/`"quota"` → 429, `"timeout"`/`"timed out"` → 504, else → **500**. This means a plain `Error` message's fate is entirely accidental: `generate-post-metadata`'s old `new Error("Post not found.")` happened to map to 404 purely because the literal substring `"not found"` appears in it, while `new Error("You do not have access to this brand project.")` and `new Error("You do not have access to this draft.")` — genuine 403 cases — contain none of the matched substrings and fell through to a fake 500, exactly as the audit suspected. `toErrorPayload(error)` always returns `{ error: <message string> }` regardless of path.

### Sweep result — every function, every expected failure mode

Converted every hand-rolled `return jsonResponse({ error: ... }, status)` and every plain `throw new Error(...)` that represented an *expected* failure mode (validation, auth, credits, forbidden, not-found) to `throw createHttpError(message, status)`, imported from `_shared/org.ts`. Left every *unexpected* failure (upstream provider misbehavior, storage upload failures, DB insert failures, unparseable LLM JSON where no dedicated handling already existed) on the generic path — those legitimately are 500s, not mis-typed ones.

## Fix 3 — metadata_status stuck-in_progress reconciliation (RELIABILITY) + ADDENDUM UPGRADE 2

**Reality check vs. audit**: Confirmed accurate — the client (`SessionStore.js`) wrote `workflow_state.metadata_status = 'in_progress'` optimistically in two places (`scheduleDraftMetadataGeneration`'s pre-write, `regeneratePostMetadata`'s pre-write) before ever invoking `generate-post-metadata`; only that edge function's success path wrote `'completed'`, and only the client's own `catch` blocks wrote `'failed'`. A lost response or closed tab between those two points left the row at `'in_progress'` forever, and `automationRunRef`'s once-per-generation-id guard meant the automatic publish-stage hydrate would never retry even on a later visit to the same generation.

### What changed — BACKEND (server owns the whole lifecycle now)

`supabase/functions/generate-post-metadata/index.ts`:
- Added a write of `workflow_state.metadata_status = 'in_progress'` + `metadata_started_at` (new key inside the existing jsonb column — **no migration needed**, jsonb is schemaless; documented per the fix's own "no migration unless you find a reason" instruction) immediately after `loadContext` resolves and prompt validation passes, whenever `body.post_id` was supplied — **before** the LLM call, not after.

On any failure there, if `body.post_id` was supplied, writes `metadata_status: 'failed'` + `metadata_error: <message>` + `metadata_updated_at` in one `UPDATE`, then re-throws so the outer catch still produces the correct typed HTTP response (Fix 2).

### What changed — FRONTEND/STORE (client stops double-writing, adds reconciliation + manual recovery)

`src/stores/SessionStore.js`:
- Removed the client-side optimistic `'in_progress'` pre-write from both `scheduleDraftMetadataGeneration` and `regeneratePostMetadata` — the server is now the only writer of `workflow_state.metadata_status`. - **Stale-`'in_progress'` reconciliation, in `hydratePostProductionFromGeneration`**: on every read, a row whose `metadata_status === 'in_progress'` is treated as `'failed'` instead if `metadata_started_at` (falling back to `metadata_updated_at`) is either missing entirely (covers rows stuck from *before* this fix existed, which never had `metadata_started_at` at all) or older than 2 minutes. ## Fix 5 — Rate limiting (previously parked) + ADDENDUM UPGRADE 3

**Context check**: read Week 1's FIXLOG "RATE LIMITING CHECKPOINT" notes first, as instructed. Only one existed (Fix 2, `enhance-prompt`'s auth fix) and it explicitly concluded no checkpoint was triggered — rate limiting was "considered and explicitly declined" as unnecessary for that fix's own correctness, not deferred with any specific numbers or design already implied. This fix starts from a clean slate rather than reconciling against prior constraints.

### DB — true sliding-window limiter, not fixed-window (per Addendum Upgrade 3)

New migration `supabase/migrations/20260711010000_edge_function_rate_limiting.sql`:
- `public.rate_limit_events (id, user_id, function_name, created_at)` — a log of individual request timestamps (not a counter-bucket), indexed on `(user_id, function_name, created_at DESC)`. RLS enabled with **zero policies** — this is the deliberate mechanism (not an oversight): RLS-enabled-with-no-policy denies every row to every non-owner role, so `authenticated`/`anon`/PostgREST's `service_role` session can never read or write this table directly — the only sanctioned path is the function below, which runs as its (owner) definer and bypasses RLS on this table by design.
- `public.check_rate_limit(p_user_id, p_function, p_max, p_window_seconds) RETURNS TABLE(allowed boolean, retry_after_seconds int)`, `SECURITY DEFINER`:
  - Counts real events within a rolling `now() - window_seconds` lookback (a genuine sliding window — recomputed fresh on every call, not a bucket that resets on a fixed clock boundary, which is exactly what avoids the 2x-burst-at-the-boundary problem fixed windows have).
  - **Concurrency safety**: `pg_advisory_xact_lock(hashtextextended(user_id || ':' || function_name, 0))` serializes the count-then-insert sequence for that exact `(user, function)` pair only — every other pair proceeds unblocked. This is a single-statement-scoped, transaction-scoped lock (auto-released when the RPC call's implicit transaction ends), closing the classic read-then-write race where two concurrent requests both observe "one under the limit" and both insert.
  - On block: computes `retry_after_seconds` as the exact time until the **oldest event still inside the window** ages out — a real number, not a guess, per the addendum's explicit requirement.
  - Opportunistic cleanup of that `(user, function)` pair's own stale rows runs inside the same locked section on every call. A separate hourly cleanup cron job (`cleanup-rate-limit-events`, 24h retention) also registered, reusing the project's **already-existing** `pg_cron` setup (confirmed via `20260710120000_vault_based_cron_secrets.sql` before writing this — did not introduce `pg_cron` as new infrastructure, per the fix's explicit instruction) to catch rows belonging to `(user, function)` pairs that simply stop being called and would never otherwise get swept by the per-call path.
  - `GRANT EXECUTE ... TO authenticated, service_role` — lets edge functions call it via either a user-scoped `authClient` or an admin client, whichever they already have on hand (see below).

### BE — shared helper + config, wired into all 9 in-scope functions

New `supabase/functions/_shared/rateLimit.ts`: `RATE_LIMITS` config map (one row per function, with the reasoning for each number written inline — reproduced in the "worst-case" table below) and `enforceRateLimit(client, userId, functionName)`, which calls `check_rate_limit` and throws a `createHttpError('Too many requests — please wait a moment.', 429)` with `.retryAfterSeconds` attached (read by `_shared/http.ts: toErrorPayload`, extended in this fix to include `retry_after_seconds` in the JSON body whenever an error carries it — additive-only, every existing consumer that only reads `.error` is unaffected) if blocked. **Fails open** (logs loudly, lets the request through) if the RPC itself errors — a broken limiter must never be the reason the whole app goes down.

Wired `await enforceRateLimit(client, user.id, "<function-name>")` immediately after `requireUser(...)` in all 9 in-scope functions: `generateImage`, `editImage`, `generateVideo`, `generate-content-plan`, `enhance-prompt`, `generate-post-metadata`, `generate-caption`, `seo-score`, `optimize-seo`. Functions that already build an admin client up front (`generateImage`, `editImage`, `generateVideo`, `generate-post-metadata`) reuse it; functions that don't (`generate-content-plan`, `enhance-prompt`, `generate-caption`, `seo-score`, `optimize-seo`) pass the existing user-scoped `authClient` instead of provisioning a new admin client just for this — `check_rate_limit` is `SECURITY DEFINER` and was granted `EXECUTE` to `authenticated`, so either client works identically.

**Deliberately excluded, per the fix's own scope**: `ai-brand-consistency-check` (already gated by its own org-credit check, `ensureCreditsAvailable`, which is a real throttle already) and `mock-publish` (the service-role scheduler path that must bypass rate limiting entirely — never touched, so it trivially bypasses since no limiter was ever added there).

### FRONTEND — friendly 429s with a real countdown (per Addendum Upgrade 3)

- `src/services/edgeFunctionClient.js: normalizeEdgeFunctionError` — added a dedicated `status === 429` branch (checked before the 401/403 branch) that reads `retry_after_seconds` from the response body and attaches it as `.retryAfterSeconds` on the returned `Error`, with the message `"You're going a bit fast — try again in a few seconds."` as a fallback if the body is somehow unreadable.
- `src/services/media.service.js: toInvokeError` — same treatment for the three media functions, which go through a separate error-normalization path. **Found and fixed a real bug while doing this**: `normalizeMediaErrorMessage` already had a heuristic that re-labels any message containing `"rate limit"` or `"429"` as a **fal.ai provider-quota-exhausted** message ("update billing or connect a media key with available quota") — without a status-code check first, our own new 429s would have been wrongly told to the user as a billing problem. Fixed by checking the real HTTP status (`error.context.status === 429`) before falling into that heuristic, routing our own rate-limit errors to their own message + real `retryAfterSeconds` instead.
- `src/pages/Studio/StudioPage.jsx` — added a small reusable countdown mechanism: `rateLimitedUntil` (per-action-key epoch-ms map) + a `setInterval` that ticks once/second while any entry is still active (to drive live re-renders) + `applyRateLimit(key, err)` (starts the countdown and toasts the exact wait time if the error carries `retryAfterSeconds`) + `getRateLimitRemaining(key)` (seconds remaining, or 0). Wired into all four client-triggered actions that call a rate-limited edge function: `handleGenerate` (key `"generate"`, also covers the carousel "Regenerate whole carousel" button since it reuses the same handler), `handleEnhance` (`"enhance"`), `handleRegenerateMetadata` (`"regenerateMetadata"`), `handleRescore` (`"rescore"`). - `PostProductionPanel.jsx` receives `metadataRetryAfter`/`seoRetryAfter` as props (computed by the parent's `getRateLimitRemaining`) and reflects them on the "Regenerate caption & title" and "Re-score"/"Retry" buttons — no separate ticking logic needed there, since the parent's own per-second re-render already flows fresh prop values down.

- **Clearing rule (decided and documented, per the addendum's explicit instruction to do so)**: called via `void clearSessionDraftPrompt(session?.id)` immediately after `ensureSession` resolves in all four generation-starting actions (`startGeneration`, `startCarouselGeneration`, `startEditGeneration`, `startVideoGeneration`) — i.e., the draft is cleared the moment a generation **starts** against that session, not when it succeeds. Rationale: attempting a generation is the point the user "used" the draft (the prompt is now embodied in a real generation attempt), regardless of whether that specific attempt succeeds or fails — a failed attempt shouldn't silently resurrect a stale draft that the user has already moved past.
- **Seeding on load**: `loadSession` (the single action both the drawer's resume flow and the URL-driven init effect funnel through, per Week 1 Fix 6) now checks, after `fetchGenerations` resolves, whether `session.metadata.draft_prompt` exists **and** the session has zero generations at all (`get().activeGenerations.length === 0` — the "no completed-generation selected-state to restore" condition from the addendum's own wording). If both hold, it calls the existing `setPromptSeed({ text, source: 'session_draft', settingsSnapshot })` — the exact same one-shot mechanism Week 1 Fix 3 built, per the addendum's explicit instruction not to invent a second seeding path.

**What changed — FRONTEND (`src/pages/Studio/StudioPage.jsx`, `PostProductionPanel.jsx` untouched)**:
- The "Save as draft without generating" button no longer disables based on `!selectedGeneration` alone — it's now only disabled when there is truly nothing to save (`!selectedGeneration && !prompt.trim()`). `onClick` branches: a selected generation still calls `saveDraft()` exactly as before (unchanged behavior); no selected generation but a typed prompt now calls the new `saveDraftPrompt(prompt)`, toasting **"Draft saved to this session"** (distinct from the generation-level "Saved as draft" message, so the two paths are never confused).
- The `promptSeed`-consuming effect (Week 1 Fix 3) gained one more branch: if the seed carries a `settingsSnapshot` (only ever true for `source: 'session_draft'` seeds), it calls `updateSettings(seed.settingsSnapshot)` in addition to restoring the prompt text — so mode, aspect ratio, slide count, batch size, video quality, etc. all come back exactly as they were when the draft was saved, not just the prompt text.

**Known, accepted minor limitation (not engineered around, low risk)**: `loadSession`'s seeding check has no "already seeded this exact session once" guard. If some future code path calls `loadSession` repeatedly for the *same* already-active, still-generation-less session without a real navigation in between, the draft would be re-seeded each time, potentially overwriting in-progress unsaved typing in the prompt box. This is not reachable through any current navigation path traced in this pass (`loadSession` is only invoked on an actual session switch via the drawer or the URL-driven init effect, both of which represent a genuine "arriving at this session" event) — flagging it rather than adding guard state for a scenario that isn't actually reachable today.

### Layers touched
Store (`SessionStore.js`), FE (`StudioPage.jsx`). No BE/edge-function changes — this is pure client+direct-Postgres-write feature work, no LLM/edge-function call involved anywhere in the save/restore path. No DB migration — `sessions.metadata` is the same pre-existing jsonb column, no new columns.

### Journey check (traced; **could not execute against a live browser/DB** — none available in this pass)
Type a prompt, switch to carousel mode and set a slide count, click "Save as draft without generating" (no generation selected) → `saveDraftPrompt` creates/reuses a session, persists `{draft_prompt, draft_settings: {contentType:'carousel', slideCount: N, ...}, draft_saved_at}` → toast "Draft saved to this session" → close the tab. Reopen later, find that session in the history drawer (it appears there since `fetchSessions` doesn't filter by generation count) → resume it → URL updates (Week 1 Fix 6) → `loadSession` fires → zero generations found → `setPromptSeed` restores both the prompt text and the carousel/slide-count settings → brief panel shows exactly what was typed and configured before. Then clicking Generate from that restored state → `ensureSession` reuses the existing (already-active) session → `clearSessionDraftPrompt` fires and removes the now-used draft from that session's metadata, so returning to it again later shows a normal (draft-cleared) session rather than re-offering the same stale draft.

## Fix 0 — deduct_credits atomicity

**Verdict: not needed.** `deduct_credits` (`20260706120000_credit_category_tracking.sql:22-57`) already uses the correct atomic pattern — a single `UPDATE ... SET balance = balance - p_amount WHERE balance >= p_amount RETURNING`, with the `credit_transactions` insert in the same function invocation. ## Fix 2 — Generation idempotency, batch failure isolation, honest cancel

### Idempotency — `request_id`/`request_slot`

**Migration** `20260712100000_week3_generation_idempotency.sql`: adds `generations.request_id text`, `request_slot integer NOT NULL DEFAULT 0`, and (formalizing pre-existing live drift already documented in the baseline migration's inline comments) `batch_id uuid`, `batch_index integer` via `ADD COLUMN IF NOT EXISTS` — safe whether or not the live table already has the batch columns. Unique index `idx_generations_request_idempotency (user_id, request_id, request_slot) WHERE request_id IS NOT NULL`. **Rollback**: drop the two new columns + the index; batch_id/batch_index are pre-existing live columns this migration only formalizes and must NOT be dropped on rollback (would destroy real carousel data).

**New shared module** `supabase/functions/_shared/generationIdempotency.ts`:
- `findCachedGeneration(adminClient, userId, requestId, requestSlot)` — looks up a `status='completed'` row at that exact key; a hit means this exact attempt already rendered and billed, so the edge function replays the cached response instead of re-rendering.
- `reserveCredits(adminClient, userId, amount, category, description)` — thin wrapper around `deduct_credits` that actually checks `ok` (the bug found in Fix 0) and throws a typed 402 on insufficient balance.
- `completeGeneration(adminClient, generationId, userId, patch)` — **the piece that makes idempotency hold even when the response never reaches the client**: writes the COMPLETED transition (merged metadata, output_url, etc.) onto the client-created PROCESSING placeholder row directly from inside the edge function, before returning. Ownership split, deliberately: `generationPipeline.js` (client) owns row *creation* and the FAILED transition (the only place that can observe "this request never reached the edge function at all"); the edge function owns the COMPLETED transition. The client's own pre-existing `.update()` to COMPLETED (in `generationPipeline.js`) was left in place as a second, redundant-but-harmless write of the same data — accepted deliberately rather than removed, since it's a real safety net if `completeGeneration`'s best-effort write silently no-ops (it fails open, logging and returning `null`, rather than ever blocking a successful render from being returned to the user).

**`generateImage/index.ts` / `editImage/index.ts` rewritten**:
- Accept `request_id`, `request_slot`, `generation_id` in the request body.
- Check `findCachedGeneration` first — a hit returns immediately with `replayed: true` and zero credits charged.
- `reserveCredits` now runs *before* prompt enhancement/provider call/upload (reserve-then-generate), with a `refundCreditsIfReserved` closure (declared outside the try block specifically so the outer catch can still call it — try/catch blocks in JS do not share `let`/`const` scope, a real bug caught and fixed while writing this) invoked on any failure after reservation.
- **Deleted the dead `record_generation` branch entirely** (not just disabled) — confirmed via 07-structural-findings.md 0.5 that every caller always passed `false`; the branch never ran in production. These functions now never INSERT a `generations` row themselves (that's `generationPipeline.js`'s job); they only `completeGeneration` (UPDATE) the row the client already created.

**Client threading** (`src/services/media.service.js`, `src/services/generationPipeline.js`, `src/stores/SessionStore.js`):
- `generateImages`/`editImage` accept `requestId`/`slotOffset`/`generationId`/`signal` and pass them through to the edge function body + `supabase.functions.invoke`'s `signal` option (confirmed supported in the installed `@supabase/functions-js` version by reading `FunctionsClient.js` directly).
- `generationPipeline.js`: `runGenerationPipeline`/`runSingleGeneration`/`runCarouselOrchestration` all accept `requestId`/`requestSlot`/`cancelSignal` and thread them into the placeholder row insert (`request_id`/`request_slot` columns) and into the `generateImage()` call (as `{requestId, requestSlot, signal, generationId}`).
- `SessionStore.js`: `startGeneration`, `startCarouselGeneration`, `startEditGeneration` each mint one `requestId = crypto.randomUUID()` per invocation (a Retry click calls the action again, getting a fresh id; nothing inside one call reuses another attempt's id) and create one `AbortController` stored in new state field `generationAbortController`.

### Batch failure isolation

- **Image batch** (`startGeneration`'s variant loop, `SessionStore.js`): previously a single variant's failure re-threw out of the whole loop, aborting all remaining variants. Now each variant iteration has its own `try/catch`, collects an `outcomes` array, and continues to the next variant on failure — mirroring the carousel path's pre-existing per-slide isolation. If `requestedVariants > 1`, `lastBatchOutcome` state is set (`{kind:'image', succeededCount, failedCount, totalCount, failedSlots, requestId}`); if ALL variants failed, still throws (nothing to show).
- **Carousel** (`generationPipeline.js: runCarouselOrchestration`): already isolated per-slide (verified in Phase 0); extended to also collect `outcomes`/`succeededCount`/`failedCount`/`totalCount` in its return value, and to check `cancelSignal?.aborted` before starting each slide (skipping — never billed, never sent to the provider — rather than attempting it).
- **UI banner** (`StudioPage.jsx`): new card shown whenever `lastBatchOutcome.failedCount > 0`, reporting "`N of M variants/slides completed`" with a "Retry failed only" button for the image-batch case (calls new `retryFailedVariants(prompt)` store action, which re-runs *only* the failed slots under the *same* `request_id` — idempotency protects the already-succeeded slots from being re-rendered/re-billed even though the retry re-invokes the whole pipeline for those slots) and a "Regenerate whole carousel" fallback for the carousel case (reuses Week 1 Fix 4's existing, already-verified full-regenerate button/handler rather than building a second, riskier per-slide-retry path for carousels in this pass — **known gap, logged below**).

**Known gap, not built**: per-slide retry-failed-only for carousels specifically (as opposed to image batches). `retryFailedVariants` as written assumes the "single" content type and would violate the new unique index if pointed at a carousel's failed slide (the slide's row already exists under that `request_id`+slot from the original carousel run; a naive re-run would try to INSERT a second row at the same key). The UI intentionally routes carousel retries to the existing, safe "regenerate whole carousel" affordance instead of a broken per-slide path. Flagging as a genuine, scoped-out follow-up rather than silently shipping a subtly-broken button.

## Fix 3 — True async video + the background job mechanism

### Scheduled-publish: deliberately NOT touched

Phase 0.3 (07-structural-findings.md) found that `process_scheduled_posts()`/`dispatch_scheduled_post()` — a real, working, idempotent (status-guarded claim), already-in-production scheduler using `pg_cron`+`pg_net` — already promotes `status='scheduled'` posts. Week 1's own FIXLOG had already discovered and documented this. Building a second `scheduled_publish` job type into the new `background_jobs` table, as the batch brief's template suggested, would have duplicated a working mechanism and recreated the exact "two writers per invariant" problem Fix 1 spent its whole effort eliminating elsewhere. **Decision, applied**: `background_jobs` in this batch is scoped to `job_type = 'video_generation'` only; the column itself is left as free text (not an enum) so a genuinely new job type can be added later without a schema migration, but nothing in this batch adds one.

### `background_jobs` table

**Migration** `20260712110000_week3_background_jobs.sql`: `id, user_id, organization_id, brand_project_id, job_type, status (queued|running|completed|failed|cancelled), payload jsonb, result jsonb, error, request_id, attempts, run_after, created_at/started_at/finished_at/updated_at`. Unique index on `(job_type, request_id) WHERE request_id IS NOT NULL` (idempotent resubmission), plus `(status, run_after)` and `(user_id, job_type, created_at DESC)` for the poller and the drawer respectively. RLS: owner (or org-brand-access, or admin) SELECT only — all writes go through `service_role`/edge functions, mirroring the `rate_limit_events` "RLS-enabled-with-no-write-policy" pattern from Week 2 Fix 5. Realtime: a `SECURITY DEFINER` trigger broadcasts every INSERT/UPDATE to a private `background-jobs-<user_id>` topic, with an RLS policy on `realtime.messages` restricting subscription to that exact user (or admin) — same private-channel pattern as Week 2 Fix 1's session broadcasts, just user-scoped instead of session-scoped. **Rollback**: `DROP TABLE IF EXISTS public.background_jobs CASCADE` (cascades the trigger; RLS policies drop with the table).

**Known, accepted scope-narrowing**: the realtime subscribe policy matches on literal user ownership of the topic string only (`topic = 'background-jobs-' || auth.uid()::text`), not org-brand-access — an org member other than the submitting user would not receive LIVE updates for an org-scoped video job via this channel (though they could still read the row directly via the table's own SELECT RLS policy, which does include org-brand-access). Not fixed in this pass since Studio (the page in scope) is a personal-workspace feature; flagged for whoever eventually wires org-workspace video generation onto this same mechanism.

### fal.ai queue extensions (`_shared/fal.service.ts`)

Added `submitVideoJob`/`getQueueStatus`/`getQueueResult`/`cancelQueueJob` (raw queue submit-with-webhook, status poll, result fetch, best-effort cancel) plus three submit-and-return wrappers (`submitVideoHailuo`/`submitVideoKling`/`submitVideoKlingI2V`) that share the exact same input-normalization logic as the pre-existing `generateVideoHailuo`/`generateVideoKling`/`generateVideoKlingI2V` (which still exist, unchanged, in case anything else ever needs the old blocking behavior — nothing in this repo does after this batch, but they weren't deleted since they're not the thing that was broken).

### `generateVideo/index.ts` — rewritten to submit-and-return

Validate → rate-limit → idempotency check (existing `background_jobs` row for the same `request_id` returns immediately, replayed) → resolve tier/credits (unchanged tier-upgrade logic) → **reserve credits atomically before submission** (deduct-at-submit, chosen over deduct-at-completion specifically because it prevents a user from queueing unlimited jobs beyond their balance while several are in flight — refunded automatically on any terminal failure) → prompt enhancement (unchanged, still synchronous/fast) → insert a `generations` row with **status `'processing'`** (fixing the video-only inconsistency the batch brief called out — video rows used to be born `'completed'`; now every media type reaches `'completed'` only via its real completion path, so Fix 1's `ensure_draft_post_for_generation` trigger behaves identically for video as for every other type) → insert a `background_jobs` row (`queued`) → submit to fal.ai's queue with a webhook URL carrying a random per-job token → update the job to `running` (best-effort; a failure here is non-fatal since fal has already accepted the job and the webhook/poller don't depend on this specific write) → return `{job_id, generation_id, status:'running', ...tier info}` immediately.

### `job-webhook` (new function) + `process-jobs` (new function) — shared finalize logic

**New shared module** `_shared/videoJobFinalize.ts`: `finalizeCompleted`/`finalizeFailed`/`reconcileJob`, used identically by both functions — one "is this job actually done, and if so, record it" implementation, not two that could drift. Design decisions:
- **Webhook body is not trusted for its shape** (fal.ai's exact webhook payload contract is not documented with enough certainty to build parsing logic around) — the webhook is treated purely as a wake-up signal; `reconcileJob` re-fetches the authoritative status/result from fal.ai's own queue endpoints, the same source of truth the poller uses. This means job-webhook and process-jobs are trivially consistent with each other by construction.
- **Auth**: a random per-job token embedded in the webhook URL (`?job_id=...&token=...`), checked against the value stored in the job's own `payload.webhook_token` at submit time. fal.ai does not document a verifiable signature scheme for this integration, so this capability-URL approach is the pragmatic choice — **flagged as unverifiable-from-this-repo**: if fal.ai's webhook payload were ever intercepted in transit (their side, not this app's), the token itself is the only protection; no additional signature check exists.
- **Webhook/poller race safety**: `finalizeCompleted`/`finalizeFailed` do the (possibly wasteful-on-a-race) fal-result fetch + storage upload FIRST, then perform the terminal status write as a single `UPDATE ... WHERE status='running'` — exactly `process_scheduled_posts()`'s claim pattern. Only the caller whose UPDATE actually matches a row (checked via `.select()`) "wins"; the loser's earlier storage upload is wasted work (an orphaned, harmless file) but its refund/generation-update side effects are correctly skipped since those only run after a confirmed win.
- `process-jobs`: sweeps `background_jobs WHERE job_type='video_generation' AND status='running' AND started_at < now() - 45s`, capped at 25/sweep, calling the same `reconcileJob`. Jobs that have been reconciled-but-still-pending 5+ times are given up on and finalized as failed-with-refund (a genuinely stuck job, e.g. fal.ai losing track of a request, self-heals into a refunded failure rather than polling forever). Registered on `pg_cron` (every minute) via `20260712120000_week3_process_jobs_cron.sql`, reusing the exact Vault-based service-role-key lookup already proven working for `process-scheduled-posts` — no new cron infrastructure introduced.

### Cancel — real, server-observed

New `cancel-video-job` edge function: verifies ownership, best-effort calls fal's queue cancel endpoint (not all models support it; failures there are swallowed internally, matching `cancelQueueJob`'s own contract), then does the same `UPDATE ... WHERE status IN ('queued','running')` claim pattern to mark the job `cancelled` and refund credits — a job cancelled here can no longer be completed by a late webhook/poll arrival even if fal renders it anyway, since `reconcileJob`'s own claim guard requires `status='running'` to finalize.
