# Stage 5c — Edge Functions: SEO Scoring, Brand Consistency, Mock Publish

## `seo-score` (`supabase/functions/seo-score/index.ts`)

- **Trigger**: `POST`, invoked by `SessionStore.js: scoreSeo` — called directly by the store action, and internally chained at the end of `optimizeSeo`.
- **Auth**: `requireUser` only; hard-requires `ANTHROPIC_API_KEY` (500 if missing, same pattern as `generate-post-metadata`/`generate-caption`) — **no credit check, no rate limit** visible for this personal-workspace call path (contrast with `ai-brand-consistency-check` below, which explicitly checks/consumes org credits — SEO scoring for personal users appears to be free/unmetered).
- **Validation**: `caption` required (400 if empty); `title`/`hashtags`/`platform`/`media_type`/`visual_prompt` all optional with defaults.
- **LLM call**: `callLlm({ preferredProvider: 'anthropic', temperature: 0.2, maxTokens: 1100, ... })` — a long, detailed "Social SEO and content-discovery strategist" system prompt (explicitly distinguishes "social discovery" scoring from "website SEO"), embeds platform-specific hashtag/tone norms, and demands a strict JSON shape with 9 breakdown dimensions, `scoreCategory`, up to 5 `recommendations`, a `benchmarkReport` array, and `hashtagSuggestions`.
- **Score normalization** (mirrors the client-side `normalizeSeoScorePayload` in `SessionStore.js` almost line-for-line — the same normalization logic exists **twice**, once server-side here and once client-side, likely because the client also needs to re-normalize whatever it reads back from `posts.seo_state` on hydration without re-calling the edge function):
  - `shouldNormalizeTenPointScale`: heuristically detects if the LLM answered on a 0-10 scale instead of the requested 0-100 (checks if max ≤10 and average ≤8.5 across all breakdown dims) and multiplies by 10 if so — a defensive correction for LLM non-compliance with the explicit "Do not use 0-10 scale" instruction.
  - Overall score: computed as a **weighted sum of the breakdown dimensions** (`keywordRelevance`/`platformFit` weighted highest at 0.16 each) if any breakdown signal exists, rather than trusting the LLM's own top-level `discoveryScore` field directly — only falls back to the raw top-level score (itself scale-corrected) if the breakdown came back entirely empty/zero.
- **DB writes**: none in this function — `posts.seo_state` is written by the **client** (`scoreSeo` action in `SessionStore.js`) after receiving this response, not by the edge function itself (contrast with `generate-post-metadata`, which does write `posts` directly) — an inconsistent pattern across these content-generation functions regarding who owns persistence.
- **Response**: massive dual-naming payload (camelCase and snake_case versions of every field) — `content_id` echoes back whichever id (`content_id`/`post_id`) was sent, `overall`/`discoveryScore`/`discovery_score` are all the same computed value.

## `optimize-seo` (`supabase/functions/optimize-seo/index.ts`)

- **Trigger**: `POST`, invoked by `SessionStore.js: optimizeSeo` — the auto-run action on entering publish stage (if caption already exists) or manually (no manual UI trigger found, per Stage 4's note).
- **Auth/dependency**: same `requireUser` + hard `ANTHROPIC_API_KEY` requirement as `seo-score`.
- **Validation**: `caption` required (400 otherwise); no `post_id`/`content_id` field in the request type at all — this function is **fully stateless**, it never looks up or writes to `posts` itself.
- **LLM call**: `preferredProvider: 'anthropic'`, `temperature: 0.3`, `maxTokens: 1300` — asks the model to both *rewrite* title/caption/hashtags for discoverability *and* self-score the result in one pass (single LLM round-trip does optimization + scoring together, unlike the client flow which calls `optimize-seo` then separately calls `scoreSeo` again afterward for the "official" post-optimization score — meaning the optimized copy gets scored **twice**: once inline by this function's own LLM call, and again by a full separate `seo-score` invocation the client triggers right after via `SessionStore.js: optimizeSeo`'s `return get().scoreSeo()` call at the end. This is a real duplicate-cost/latency inefficiency: two separate LLM calls where the second one's score could arguably have reused the first call's self-reported score instead).
- **Hashtag platform rules** (`normalizeHashtags`): caps at 2 for `twitter`/`x`, 10 otherwise — enforced in code regardless of what the LLM returns, a harder guarantee than the system prompt's soft instructions.
- **Score normalization**: same 0-10-vs-0-100 scale-correction heuristic pattern as `seo-score`, but simpler (single top-level `discoveryScore`/`seoScore` field, not the weighted-breakdown recomputation `seo-score` does) — `optimize-seo`'s own returned `seoScore`/`discoveryScore` is therefore **not** computed the same way as `seo-score`'s `overall`, another source of the "two different scores for the same content" duplication noted above, compounding the double-LLM-call issue since even the two scores computed for nominally the same optimized copy can differ due to different normalization math, not just different LLM sampling.
- **DB writes**: none — same "stateless, client persists the result" pattern as `seo-score`.
- **Response**: `optimizedTitle`/`optimizedCaption`/`optimizedHashtags` (+ snake_case duplicates), full score breakdown/benchmark/hashtag-suggestion payload (+ duplicates), `improvements`/`improvementReport`/`recommendations` (several overlapping array fields derived from overlapping LLM response keys via fallback chains — `normalizeImprovementReport` falls back through `improvementReport → improvement_report → benchmarkReport → benchmark_report → improvements` if the model didn't populate the exact expected key, a defensive-parsing pattern repeated across all these SEO functions).

## `ai-brand-consistency-check` (`supabase/functions/ai-brand-consistency-check/index.ts`)

- **Trigger**: `POST`, invoked by `SessionStore.js: checkBrandConsistency(organizationId, brandProjectId)` — **org-workspace only**; per Stage 3b/4's notes, no call site was found in the reviewed `StudioPage.jsx`/`PostProductionPanel.jsx` for this page's personal-workspace UI, so this function's actual invocation from *this specific page* is UNCLEAR/likely not exercised by the current Generate/Studio UI (it may be wired into an org-specific post-production variant not covered by this audit's file set).
- **Auth + authorization**: `requireUser`, then `requireActiveOrgMember(adminClient, organization_id, user.id)` (must be an active member of the org) **and** `ensureBrandProjectAccess(member, brand_project_id)` (member must specifically have access to that brand project, not just the org generally) — a stricter two-tier check than most other functions in this audit.
- **Credit gating**: `ensureCreditsAvailable(adminClient, organization_id, member, 1)` — **pre-flight credit check specific to the org's credit pool** (distinct from the personal `user_credits` table checked by the media-generation functions), throws before any LLM call if insufficient. After a successful call, `recordCreditUsage(...)` records `1` credit consumed under `eventType: 'consistency_check'`. This is the **only** function in this whole audited set that both checks *and* records org-level credit consumption for a text/LLM-only operation — `seo-score`/`optimize-seo`/`generate-post-metadata`/`generate-caption` are all unmetered for any workspace.
- **Brand voice sourcing**: fetches the org + brand project row, then `getBrandKitSystemPrompt(adminClient, brand_project_id)` (a pre-built stored prompt) with a client-constructed fallback (`buildBrandKitSystemPrompt`) from the brand project's raw `brand_settings` JSON if no stored prompt exists.
- **LLM call**: `preferredProvider: String(organization.settings?.ai_model || 'anthropic')` — **the only function in this whole set whose provider preference is configurable per-organization** (via `organization.settings.ai_model`) rather than hardcoded in the function itself; `jsonMode: true`, `temperature: 0.2`, asks for `{ overall_score, passes, issues[], positive_notes[] }`.
- **Parse failure handling**: unlike every other SEO/metadata function, this one does **not** use the defensive `pickJson`-style substring extraction — it calls `JSON.parse(result.content)` directly, and if that throws, converts it into a clean `createHttpError('AI response could not be parsed as JSON.', 500)` rather than letting a raw `SyntaxError` propagate — slightly better error messaging than some siblings, but more brittle to the model wrapping its JSON in prose/fences (no fence-stripping here at all).
- **DB writes**: none to `posts`/brand tables — only the credit-usage ledger write via `recordCreditUsage`.
- **Response**: `{ result: <parsed LLM JSON>, credits_consumed: 1, model }`.

## `mock-publish` (`supabase/functions/mock-publish/index.ts`)

- **Trigger**: `POST`, invoked once per selected account by `mockPublishService.js: publishPost`, itself called sequentially per attempt by `mockPublishWorkflow.js: executeMockPublishAttempts` during `publishContent`'s immediate-publish path.
- **Dual auth modes**: supports being called either as a normal authenticated user (`requireUser`) **or** as a trusted service-role caller (`isServiceRoleRequest` checks the `Authorization` header for an exact match against `SUPABASE_SERVICE_ROLE_KEY`, then `requireServiceRole(req)` — presumably for a server-side scheduler/cron process publishing *scheduled* posts without a live user session, though that caller is out of scope for this page's own client code, which always goes through the normal-user path).
- **Validation**: `post_id`+`connected_account_id` required (400 otherwise).
- **Authorization** (only enforced when `requesterId` is set, i.e., not a service-role call): if the post is org-scoped, requires active org membership; if personal, requires **both** `post.user_id === requesterId` **and** `account.user_id === requesterId` (double ownership check — the post and the target account must both belong to the caller). Also separately validates `account.scope === 'organization'` implies `account.organization_id === post.organization_id` (prevents publishing a personal post through an org-scoped connected account or vice versa, or through an org account belonging to a *different* org).
- **Idempotency**: if a `publish_request_id` was supplied (always the case from this page's client, which generates one via `buildPublishRequestId`), it first checks `mock_publish_logs` for an existing row matching `(post_id, connected_account_id, publish_request_id)` — if found, **returns the cached prior result directly without re-running the mock publish**, making repeated/retried calls with the same request id safe and idempotent (this is a real, working idempotency mechanism — contrast with the client-side `inFlightPublishRequests` Set in `mockPublishWorkflow.js`, which only dedupes truly *concurrent* in-flight calls within the same browser tab; this server-side check additionally protects against genuinely separate/sequential retries, e.g., a page refresh followed by re-clicking Publish).
- **Actual "publish"**: delegates to `runMockPublish` (`_shared/mockPublish.ts`, not read in this stage — out of scope, but understood by name/usage to be the simulated-provider logic that presumably writes the `mock_publish_logs` row, decides success/failure, generates a fake `mockPostId`/`mockPostUrl`) using the generation's `storage_path` as the media URL.
- **DB reads**: `connected_accounts` (full row), `posts` joined with `generations(storage_path, media_type)`.
- **Error handling**: `createHttpError` used consistently for expected failure modes (404 not-found, 403 forbidden, 400 mismatched org scope) — the most consistently-typed error handling of any function reviewed in this stage; falls through to the generic `toErrorPayload`/`mapErrorToStatusCode` catch-all for anything unexpected (including whatever `runMockPublish` itself might throw).
- **This is explicitly a simulated/mock publish, not a real social-platform API integration** — confirmed by the function name, the `mock_publish_logs` table name, and the client-side UI copy ("Simulated publish" badge in `PostProductionPanel.jsx`, "Publish now (simulated)" toast messages) — no real Instagram/TikTok/etc. API call happens anywhere in this traced flow.
