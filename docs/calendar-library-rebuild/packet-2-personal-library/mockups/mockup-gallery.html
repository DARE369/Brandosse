Upload flow · spec §5

Non-blocking duplicate warning

Checksum computed immediately on upload completion; if it matches an existing asset, a non-blocking "this looks like a duplicate of [X]" warning appears before the user finishes tagging — never blocks the upload. The user can confirm "this is a new version" (creates a version-history link, §6.2) or dismiss it as an intentional separate re-upload.

Detail · spec §6.2

Version history — superseded-version chain

Re-uploading "the same" asset (matched via duplicate-detection, user confirms "this is a new version") creates a new row and links it via superseded_by_asset_id. The old row is hidden from default grid/table views but stays fully visible and clickable here, and remains in "used in" history on any post that referenced it — directly addressing the "published the wrong/outdated version" pain point.

Detail · spec §6

Soft-delete with recovery-window messaging

Delete moves an asset to a recoverable trash state, not an immediate hard delete — a defined recovery window is communicated explicitly at the moment of deletion. Reliability requirement: an accidental delete of a brand's only logo file should never be unrecoverable.

Click Delete to see the confirmation + recovery messaging

Cross-link · spec §7 — the highest-value connection across both specs

"Schedule" hand-off into the Calendar's Quick Post composer

Every asset's "Schedule" action opens the Calendar spec's shared ScheduleModal / QuickPostComposer, pre-filled with this asset — the same shared component, not a Library-specific reimplementation. This is mocked as a genuine click-through between the two standalone mockup galleries (not described, not simulated in-page): the link below opens ../../packet-1-personal-calendar/mockups/mockup-gallery.html#quick-post in a new tab, which is Packet 1's real Quick Post composer section — its first step is "Library asset (optional)" with an Asset Picker modal, the closest existing anchor to "a composer that accepts a Library asset." See DECISIONS_LOG.md for why this exact anchor was chosen over building a new modal state.
