**2026-06-25T02:25:00 — `implementation-researcher`**
**Decision:** Recommended routing the new Library upload through a new Supabase Edge Function modeled on `org-asset-upload/index.ts`'s shape (server-side MIME/size validation, admin-client storage write, row insert inside the trusted function) rather than continuing personal's current client-direct-to-storage pattern (`LibraryStore.uploadMediaAsset()`), even though this means a heavier Phase 3 lift than "just add a checksum field to the existing client-side flow."
**Reasoning:** `LIBRARY_SPEC.md` §10 explicitly requires server-side-enforceable file validation as a hook point ("mime-type allowlist, size cap, dimension/duration sanity checks... this should be added here as a hook point even if the actual scanning service is chosen later"), and client-side-only validation (the `accept` attribute pattern, or a client-side size check before upload) is advisory only — trivially bypassed by any caller that isn't the one UI that added the check. `org-asset-upload` is the only one of the four existing upload paths that actually satisfies this today, and it's a real, proven, already-deployed pattern in this exact codebase, not a new architecture being proposed from nothing.
**If wrong:** If the human reviewing this research decides client-side-only validation is an acceptable trade-off for personal specifically (lower implementation cost, personal data has no confidentiality requirement the spec names, unlike org's client-confidential-material concern), this recommendation should be revisited at the mockup/Phase-3-design stage — the mockup itself doesn't change either way (the user-facing upload flow looks identical whether validation happens client-side or server-side), so this decision can be deferred without blocking Phase 2 if needed.

**2026-06-25T02:35:00 — `implementation-researcher`**
**Decision:** Recommended a two-tier duplicate-detection approach (SHA-256 exact hash + a Canvas-native perceptual/average-hash for images), both computed **client-side**, explicitly rejecting the more "obvious-by-precedent" option of computing the perceptual hash server-side in a new edge function alongside the upload validation step recommended above.
**Reasoning:** Web research surfaced a concrete, codebase-specific blocker: Supabase Edge Functions run on Deno and only support WASM-based image-processing libraries, not native-binding libraries like `sharp` that the common Node.js perceptual-hashing path (`sharp-phash` and similar) depends on. A correct server-side perceptual hash in this runtime would require a WASM image-decoding library (e.g. `magick-wasm`) doing manual decode/resize/greyscale/DCT steps — a heavier build than this v1, non-blocking-warning feature justifies. A `Canvas`-based average-hash runs with zero new dependencies (browser-native `HTMLCanvasElement`/`getImageData()`) and is sufficient for the spec's stated case (resized/recompressed edits of an image). This is one of the few places in this research where "do it the way everything else in this codebase does it (server-side, in an edge function)" and "do it the way that's actually easy and dependency-free in this specific runtime" diverge — flagged explicitly as a deliberate judgment call rather than silently picking the pattern-matching answer.
**If wrong:** If a future version of Supabase Edge Functions adds native-binding support, or if a suitable WASM perceptual-hash library is found that's lighter than expected, this could move server-side without changing any user-facing behavior (the warning still just says "looks like a duplicate of X") — so the cost of being wrong here is a Phase 3 implementation-location change, not a design rework. If client-side canvas hashing turns out to have its own cross-browser inconsistency problems (e.g., colour-management/EXIF-orientation differences producing different hashes for the same visual image across browsers), that would need to surface as a bug report post-implementation, not something this research could fully verify without building and testing the actual hash function — flagged as residual risk rather than a confirmed problem.

**2026-06-25T02:45:00 — `implementation-researcher`**
**Decision:** Recommended storing two distinct hash columns (an exact-hash `checksum` plus a separate `perceptual_hash`) on the new `assets` table, diverging from `LIBRARY_SPEC.md` §2.1's literal schema, which names only a single `checksum` field described as "perceptual/content hash."
**Reasoning:** The spec's own field description conflates two different things that the research in §2 establishes need to be computed and compared differently (exact-match lookup vs. Hamming-distance-threshold scan) — storing them as one column would force picking only one algorithm and lose the other case entirely (a perceptual hash can't detect byte-identical duplicates any better than visually-different files, and an exact hash by definition can't catch resized/recompressed edits). Flagging the spec's own field list as needing a small addition is exactly the kind of finding Phase 1 research is supposed to surface before Phase 2/3 lock in a schema that can't actually do what §5/§6.2 ask of it.
**If wrong:** If the human prefers a single combined field for schema simplicity (e.g., storing both hashes concatenated in one text column and parsing at query time), that's a viable alternative with the same net capability — this is a minor schema-shape preference, not a capability disagreement, and easy to change before any migration is written since no implementation has started.

**2026-06-25T10:05:00 — `feature-data-layer-builder`**
**Decision (THE architecturally significant call):** The new personal asset table (`personal_assets`) will hold a **real, independent row for every source** — `upload`, `generation`, and `post` — not just uploads. For `source = 'generation'`/`source = 'post'` rows, `generation_id`/`post_id` are populated as provenance FKs, but the row's descriptive/administrative fields (`title`, `tags`, `ai_tags`, `alt_text`, `checksum`, `status`, `used_in_post_ids`, `superseded_by_asset_id`, soft-delete) live independently on the new table, not derived live from `posts`/`generations` at read time. Population for `source='generation'`/`source='post'` rows happens via a **new, additive trigger** (`sync_personal_asset_from_post()`, `AFTER INSERT OR UPDATE ON public.posts`) that upserts into `personal_assets` — a parallel mechanism that runs *alongside* `ensureLibraryRowsForPosts()`/the existing `create_library_item_from_post` trigger, never replacing or calling them. I explicitly rejected the alternative of a read-time UNION/view over `posts`+`generations`+`media_assets` (uploads-only-table + view-time join for the other two sources).
**Reasoning:** This is the load-bearing decision the task asked me to make and justify. Walking through why a real-row-per-source model is the only one that satisfies every constraint simultaneously:
1. **The mockup's own requirements are the test.** Every card in `mockup-gallery.html` — regardless of source badge (`Upload`/`Generation`/`Post-linked`) — supports: inline-editable `title`/`description`/`alt_text` (lib-meta-grid, all three demo cards in the grid use the same drawer), user `tags` distinct from `ai_tags` (`asset-card__tag` vs `asset-card__tag.is-ai`, shown on a generation-sourced card "Launch announcement render" exactly as on the upload-sourced "Studio shot" card), a `used_in_post_ids`-driven "Unused" badge/filter (Card 6, "Product demo clip," is `source-generation` AND carries the `Unused` badge — meaning a generation's "unused" status must be a real, queryable, independently-updatable fact, not something inferred fresh from a join every time), a version-history chain (`superseded_by_asset_id` — the spec doesn't scope this to uploads only, and nothing in the mockup suggests generation-sourced assets are exempt from re-upload/supersede), and soft-delete with a recovery window. None of `posts` or `generations` has columns for `ai_tags`, `alt_text`, `checksum`, `superseded_by_asset_id`, or a soft-delete state — and they should not gain them, because those are Library-specific concepts that would pollute the Generate Studio/post-authoring schema Master Brief §0 rule 2 puts off-limits. A view can join columns that exist; it cannot manufacture columns that don't exist anywhere to be joined, and it cannot be written to (you cannot `UPDATE` a plain view's joined-in `ai_tags` column when there is no `ai_tags` column on the underlying `posts`/`generations` row at all).
2. **Editable, source-independent metadata requires a real row to hold the edit.** The drawer's "Save changes" button (every source, same drawer markup) must persist a human-edited `title`/`description`/`alt_text`/`tags` somewhere. For an `upload`-sourced asset that's obviously the new table's own row. For a `generation`-sourced asset, the only structurally honest place to persist a Library-specific edit (e.g., the user retitles a generation's Library card) without writing it onto the `generations`/`posts` row itself (out of scope, Master Brief §0 rule 2 — Generate Studio's own data must not be touched by Library) is a real row on the new table that *references* the generation but owns its own Library-facing fields. A view-based approach would have nowhere to write that edit at all, or would be forced to write it back onto `generations`, which is the one thing this task explicitly says not to do.
3. **`used_in_post_ids` must be an independently-queryable, independently-updatable fact, not a derived join.** The spec's own §2.1 names this as an array column precisely because the "Unused" filter and the detail drawer's "Used in" list both need direct, indexable access to it (RESEARCH.md §3.1 already flagged this exact distinction against org's `usage_count` integer). A `post`-sourced asset's "used in" list is trivially the post it came from, but a `generation`-sourced asset can be used on *multiple* posts over time (the same generation reused), and an `upload`-sourced asset's usage is something only Library itself can ever know about (no other system writes it). Maintaining one real array column on one real row, updated by Library's own write paths (and, additively, by a small trigger keyed off `posts.generation_id`/`posts.id` insert/update — see below), is the only way to make "is this unused" a single indexed boolean-equivalent check across all three sources rather than three different queries with three different reliability profiles depending on source.
4. **The non-negotiable constraint — never touch `ensureLibraryRowsForPosts()` or its call sites — is fully respected either way, but the real-row model is the *cleaner* way to respect it.** Both options (real-row-per-source vs. read-time UNION) can be built without touching that function. But the read-time UNION option would still need *something* to keep `media_assets`-equivalent uploads-only data combined with live `posts`/`generations` reads, and critically, it would need a separate, bolted-on side-table just to hold the editable/soft-delete/version fields for `generation`/`post` rows anyway (per point 2) — meaning the "simpler" read-time UNION doesn't actually avoid building a real per-source row; it just splits the row into two tables joined awkwardly (one for provenance, one for editable Library metadata) instead of one. That's strictly worse: two tables to keep in sync with each other PLUS a view to join them, versus one table and one additive trigger.
5. **Population mechanism — a new, additive `AFTER INSERT OR UPDATE` trigger on `posts`, not a second client-side JS call alongside `ensureLibraryRowsForPosts()`.** The existing `create_library_item_from_post` trigger (audit §0) already proves this codebase's own established pattern for "keep a junction/derived table in sync with `posts` writes without relying on every call site remembering to call a JS helper" — a DB trigger fires unconditionally on every insert, including the admin-moderation and SessionStore paths the audit found, and any future write path nobody thinks to wire a JS call into. A trigger is strictly more reliable here than asking a future engineer to remember "also call `syncPersonalAssetFromPost()` next to `ensureLibraryRowsForPosts()`" at 6 call sites forever. The trigger only fires for **personal-scope posts** (`organization_id IS NULL`, matching `PERSONAL_WORKSPACE_SPEC.md`'s scoping convention) — explicitly closing the cross-contamination gap AS_IS_AUDIT.md §0/§7.1 flagged for `content_library_items` (an org-scope draft will NOT create a stray `personal_assets` row, unlike the existing junction-table behavom this new table deliberately does not replicate).
6. **Generation-sourced rows specifically:** populated by the same trigger, reading `NEW.generation_id` when present and `NEW.generation_id IS NOT NULL` — one `personal_assets` row per distinct `generation_id` per user (not one per post), upserted (`ON CONFLICT (user_id, generation_id) WHERE generation_id IS NOT NULL DO NOTHING` for the row's existence, with `used_in_post_ids` appended via a separate `array_append`-if-not-present update) so that a generation reused across multiple posts/drafts gets exactly one Library card with a growing `used_in_post_ids` list — matching Card 2 in the mockup ("Launch announcement render," `used ×3`) exactly. Post-sourced rows (no `generation_id`, e.g. a pure Quick-Post-with-no-asset draft) get their own `personal_assets` row keyed by `post_id` instead, one-to-one.
**If wrong:** If a future review decides the per-source real-row model is too heavy (e.g., storage/row-count concerns at scale, or a preference for the new table to be uploads-only with org-style separate junction tables for the rest), the fix is structural but contained: the trigger can be dropped and replaced with a view, and the editable-metadata problem (point 2 above) would need its own resolution at that time (e.g., a separate `personal_asset_overrides` table keyed by `generation_id`/`post_id`, which is exactly the "two tables instead of one" shape this decision argues against as strictly worse, but would not be a wasted rebuild — the upload-sourced rows and all RLS/edge-function work below are unaffected either way, only the generation/post sync mechanism would need to change). The trigger itself is additive and reversible — dropping it does not corrupt or require migrating any existing data, since `personal_assets` rows for `generation`/`post` sources are themselves trigger-maintained, not hand-entered.
