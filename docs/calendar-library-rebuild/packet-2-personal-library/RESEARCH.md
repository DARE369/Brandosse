### 1.2 The progress mechanism worth reusing directly: `uploadWithProgress()`

`src/stores/BrandKitStore.js:32-78` (`uploadWithProgress(bucket, storagePath, file, onProgress)`) solves a real, non-obvious problem: the Supabase JS client's `storage.from(bucket).upload()` method does not expose upload-progress events. This function works around that by hand-rolling the multipart PUT as a raw `XMLHttpRequest` against Supabase Storage's REST endpoint directly, attaching `xhr.upload.onprogress` to compute a percentage and call the caller's `onProgress` callback:

```js
const uploadUrl = `${SUPABASE_URL}/storage/v1/object/${bucket}/${encodeStoragePath(storagePath)}`;
xhr.open('POST', uploadUrl, true);
xhr.setRequestHeader('Authorization', `Bearer ${session.access_token}`);
xhr.setRequestHeader('apikey', SUPABASE_ANON_KEY);
...
xhr.upload.onprogress = (event) => {
  if (!event.lengthComputable) return;
  const pct = Math.round((event.loaded / event.total) * 100);
  if (typeof onProgress === 'function') onProgress(pct);
};
```

This is the single most directly-reusable piece of code found for this packet's upload flow. `uploadAsset()` (`BrandKitStore.js:253-311`) wraps it in a clean insert-row-first ŌåÆ upload ŌåÆ update-status-on-completion-or-failure sequence (`ASSET_STATUS.UPLOADING` ŌåÆ `READY`/`FAILED`), which is exactly the per-file status machine ┬¦5/┬¦11's "upload validation failure... doesn't block the rest of a multi-file batch" requirement needs. **Recommendation: the new Library upload flow's data layer should adapt `uploadWithProgress()` + this insert-then-upload-then-finalize sequencing directly**, parameterized for the new `assets` table/bucket rather than `brand_assets`, instead of writing a new progress mechanism from scratch or accepting `LibraryStore.uploadMediaAsset()`'s current no-progress, no-status-machine, direct-`supabase.storage.upload()` approach (AS_IS_AUDIT.md ┬¦1.5, confirmed again directly at `LibraryStore.js:335-338`).

### 1.3 The server-side validation gap, and which existing pattern closes it

AS_IS_AUDIT.md ┬¦2/┬¦5 already flagged that `LibraryStore.uploadMediaAsset()` has zero validation of any kind ŌĆö confirmed again here by direct re-read (`LibraryStore.js:314-426`, the only guard is `if (!file) throw new Error(...)`). `LIBRARY_SPEC.md` ┬¦10 requires mime/size/dimension validation as a hook point. Two real validation patterns already exist to choose between:

- **Client-side only** (`AssetUploader.jsx`'s `accept` attribute) ŌĆö cheap, but advisory only; trivially bypassed, and provides no defense if the upload path is ever called from anywhere other than this one UI (e.g., a future API or bulk-import path).
- **Server-side, in an edge function** (`supabase/functions/org-asset-upload/index.ts`) ŌĆö `inferFileType(file)` (`:57-64`) derives `file_type` from the real `File.type` MIME string server-side rather than trusting client input, and the function holds the only code path that writes the `org_asset_library` row at all ŌĆö meaning a client cannot insert a fabricated row bypassing validation, because RLS on `org_asset_library` requires `uploaded_by = auth.uid()` but the actual file-type inference and row-shape construction happens inside the trusted edge function, not client-side.

**Recommendation: route the new Library upload through a new edge function (e.g. `personal-asset-upload`), following `org-asset-upload`'s shape** (`FormData` in, server-side MIME inference + size check, storage upload via the admin client, then the `assets` row insert) **rather than continuing `LibraryStore.uploadMediaAsset()`'s pattern of uploading directly from the browser with the anon key and inserting the row client-side.** This is the only one of the four existing upload paths that actually satisfies ┬¦10's intent today, and it is a real, working, already-proven pattern in this exact codebase ŌĆö not a new architecture being proposed from nothing. The trade-off: an edge function adds a network hop and means progress-reporting must come from the client's `XMLHttpRequest` to the edge function's URL instead of directly to Supabase Storage's REST endpoint ŌĆö `uploadWithProgress()`'s technique (raw XHR with `upload.onprogress`) still applies, just pointed at the edge function's URL instead of the storage REST URL directly, since edge functions support streaming `FormData` uploads and `xhr.upload.onprogress` fires for any XHR request body regardless of destination.

### 1.4 Accessibility findings for the upload UI specifically (web research)

- A file upload control must have a clear, associated `<label>`, plus `aria-describedby` linking to format/size-limit helper text, and `aria-invalid` + linked error text on validation failure ŌĆö drag-and-drop must never be the only path; a standard file input fallback must always exist alongside any custom dropzone, and a custom dropzone needs `tabindex="0"` + `role="button"` + a descriptive `aria-label` if it isn't a native `<label>`/`<input>` pairing. ŌĆö [HTML File Upload Accessibility with WCAG and ARIA Best Practices, Filestack](https://blog.filestack.com/html-file-upload-accessibility/)
- Upload progress must be exposed through `aria-live="polite"` regions so screen reader users get the same "uploading / complete / failed" feedback sighted users get from a visual progress bar, and each file in a multi-file batch should be treated as its own labeled, individually-announced unit rather than one bulk status ŌĆö directly relevant to ┬¦5's "per-file progress" and ┬¦11's "doesn't block the rest of a multi-file batch" requirements, which need to be screen-reader-legible per-file, not just visually per-file. ŌĆö [HTML File Upload Accessibility with WCAG and ARIA Best Practices, Filestack](https://blog.filestack.com/html-file-upload-accessibility/)
- `AssetUploader.jsx`'s dropzone already does the keyboard-accessible part correctly today ŌĆö `role="button"`, `tabIndex={0}`, `onKeyDown` handling Enter/Space to trigger the hidden file input (`:134-141`), and a real `aria-label="Upload brand assets"` (`:142`). This is a legitimate, already-built accessible-dropzone pattern in this codebase and should be the basis for the new Library upload dropzone's markup, not reinvented. What it's missing relative to the Filestack guidance above: no `aria-live` region for upload status (the `uploadItems` progress list at `:166-180` is purely visual, no `aria-live`/`role="status"` anywhere), and no `aria-describedby` linking the dropzone to format/size-limit text. Both are additive fixes to an already-sound base, not a rebuild.

## 2. Perceptual-hash / duplicate-detection approach

### 2.1 What's already a dependency ŌĆö checked first, per instruction

Re-confirmed directly from `package.json:27-67` (full dependency + devDependency list, read in this session): **no hashing, checksum, or image-processing library of any kind is present** ŌĆö no `crypto-js`, no `blockhash`, no `phash`/`sharp`/`jimp`/`image-hash`, nothing in that family. The only file-adjacent libraries present are `axios` (HTTP), `@supabase/supabase-js` (storage client), and `zod` (validation, schema not file-content). This confirms AS_IS_AUDIT.md's adjacent finding (no checksum computation anywhere in `LibraryStore.js`) at the dependency level too: there is genuinely nothing to reuse, this is a from-scratch decision either way.

The only existing "hash" in the codebase, `src/utils/brandKitHash.js`'s `computeBrandKitHash()`, is a trivial 32-bit string hash (Java's classic `hashCode()` algorithm) over a few brand-kit text fields ŌĆö not file-content hashing, not cryptographic, not relevant precedent for this question.

### 2.2 Exact-hash vs. perceptual-hash, and where each is computed

**Cryptographic (exact) hash** (e.g. SHA-256 via the browser's native `SubtleCrypto` API): two files produce the same hash only if they are byte-for-byte identical; a single re-compression, resize, or even a metadata-only change (different EXIF data from a re-export) produces a completely different hash, by design ŌĆö "the avalanche effect of small changes in input leading to drastic changes in the output." ŌĆö [Scale-Ready: Image Deduplication with Cryptographic & Perceptual Hashes, Towards Dev](https://medium.com/towardsdev/scale-ready-image-deduplication-with-cryptographic-perceptual-hashes-ae363f54710f)

**Perceptual hash** (e.g. average-hash/aHash, difference-hash/dHash, or pHash/DCT-based): designed so visually similar images land close together in hash-space (measured by Hamming distance) even after resizing, re-compression, or minor edits ŌĆö "perceptual hashes are 'close' to one another if the features are similar... resilient to minor alterations such as resizing, brightness shifts, and compression, making it better suited for detecting near-duplicates." A similarity threshold (commonly up to Hamming distance 5, though implementation- and corpus-dependent) is needed to decide "duplicate enough to warn," since perceptually-identical images rarely hash to a distance of exactly 0. ŌĆö [Duplicate image detection with perceptual hashing in Python, Ben Hoyt](https://benhoyt.com/writings/duplicate-image-detection/); [Scale-Ready: Image Deduplication, Towards Dev](https://medium.com/towardsdev/scale-ready-image-deduplication-with-cryptographic-perceptual-hashes-ae363f54710f)

**The spec's own framing already names the right answer for v1, and codebase-fit research confirms it rather than overturning it.** `LIBRARY_SPEC.md` ┬¦5/┬¦6.2 explicitly wants to catch "an edited version" of an existing asset (re-compressed, resized, lightly cropped/touched-up) ŌĆö this is precisely the case exact-hash *cannot* catch (any pixel change at all produces a fully different SHA-256) and perceptual hash is designed for. But there's a real implementation-location complication specific to this stack, found via web research, that the spec doesn't address and Phase 2/3 needs to know about before committing to "perceptual hash, full stop":

### 2.3 The Supabase Edge Functions constraint that affects where the hash gets computed

Per Master Brief's framing that Supabase Edge Functions are already used elsewhere in this repo (true ŌĆö 50 functions, confirmed via `Glob` on `supabase/functions/*/index.ts`), the natural assumption is "compute the perceptual hash server-side, in an edge function, alongside the new upload edge function recommended in ┬¦1.3." Web research found a concrete blocker for that specific placement:

> "Supabase Edge Functions currently doesn't support image processing libraries such as Sharp, which depend on native libraries. Only WASM-based libraries are supported... The recommended approach uses magick-wasm... Edge Functions supports running WebAssembly (Wasm) modules" ŌĆö [Image Manipulation, Supabase Docs](https://supabase.com/docs/guides/functions/examples/image-manipulation); corroborated by [Using Wasm modules, Supabase Docs](https://supabase.com/docs/guides/functions/wasm)

This means the easy, common Node.js path for perceptual hashing ŌĆö `sharp`-based libraries (e.g. `sharp-phash`) ŌĆö **does not run in this codebase's Deno edge function runtime at all**, because `sharp` depends on native (non-WASM) bindings. A perceptual hash computed server-side in this stack would require either (a) a WASM-compiled image library (`magick-wasm` or similar) doing the decode + resize + greyscale + DCT/average steps by hand, which is meaningfully more edge-function code and CPU time than this v1 feature warrants, or (b) shelling out to an external service, which the spec doesn't ask for and which would introduce a new infra dependency this packet has no mandate to add.

**Browser-side perceptual hashing, by contrast, has a real lightweight option already proven to run client-side without native bindings:**

> "pHash provides perceptual hashing and visual comparison of images in the browser without using HTML canvas" ŌĆö [phash-js, GitHub](https://github.com/freearhey/phash-js); "Blockhash is a perceptual image hash calculation tool... can be used in the browser" ŌĆö [blockhash-js, GitHub](https://github.com/commonsmachinery/blockhash-js)

A `Canvas`-based average-hash/difference-hash (draw the image to an offscreen `<canvas>` at a small fixed size, e.g. 8x8 or 16x16, read pixel data via `getImageData()`, threshold against the average luminance, pack into a bit-string) is implementable with **zero new dependencies** ŌĆö `HTMLCanvasElement` and `CanvasRenderingContext2D` are browser-native APIs, no library install required at all ŌĆö which is a stronger fit for the "don't add a new dependency without checking" instruction than even installing `blockhash`/`phash-js` would be.

### 2.4 Net recommendation

**Two-tier approach, computed entirely client-side, no new dependency:**

1. **Exact hash (SHA-256 via `crypto.subtle.digest()`)** ŌĆö computed first, immediately on file selection, before upload starts. Browser-native, no dependency (`SubtleCrypto` API; confirmed feasible and already a documented pattern ŌĆö "the checksum computed entirely on the client side... using the browser's FileReader API in conjunction with the SubtleCrypto API to read the file in chunks (streaming) and progressively calculate the hash" ŌĆö [SHA-256 in JavaScript in Browser, ssojet](https://ssojet.com/hashing/sha-256-in-javascript-in-browser); [How to compute SHA hash of local file in Javascript using SubtleCrypto API, TechOverflow](https://techoverflow.net/2021/11/26/how-to-compute-sha-hash-of-local-file-in-javascript-using-subtlecrypto-api/)). This catches the cheap, common case (re-uploading literally the same file) at near-zero cost and is the value stored in `assets.checksum` for any exact-match lookup.
2. **Perceptual hash (Canvas-based average-hash, image files only)** ŌĆö computed client-side immediately after, draws the image to a small offscreen canvas, derives a fixed-length bit-string, compared against existing assets' stored perceptual hashes via Hamming distance for the "this looks like a duplicate of [X]" warning (┬¦5) that needs to survive resize/recompression, not just byte-identical re-uploads. Video files fall back to exact-hash-only (no perceptual-hash check) for v1 ŌĆö perceptual video hashing (frame sampling + per-frame hashing) is a materially bigger feature and isn't named in ┬¦5's scope, which talks about "resized/re-compressed versions of the same image," not video.
3. **Both hashes travel with the upload request** (computed before or during the upload, not after) and get stored on the new `assets` row; the duplicate-check query (exact match on `checksum`, Hamming-distance-under-threshold scan on the perceptual hash among the user's own assets) runs server-side in the upload edge function recommended in ┬¦1.3, since it needs to compare against other rows' stored hashes ŌĆö a comparison the browser can't safely or efficiently do as the asset library grows.

**Why not push the perceptual hash computation server-side instead:** the Sharp/native-library limitation (┬¦2.3) makes that the harder, heavier path in this specific runtime, for a feature whose own spec describes it as a "non-blocking warning," not a hard gate ŌĆö the cost/value ratio favors computing it where it's cheap (client, canvas-native) over where it's spec-symmetric-feeling but actually harder to implement (edge function, WASM image library). This is the first place in this packet's research where the "what's idiomatic in this codebase" instinct (do it server-side, like everything else) and the "what's actually easy in this runtime" finding (do it client-side) point in different directions ŌĆö flagging this explicitly as a deliberate judgment call, not an oversight.

**Open question for the mockup/implementation phase, not resolved here:** exact perceptual-hash *algorithm* choice (average-hash vs. difference-hash vs. a small block-hash) and the Hamming-distance threshold for "warn" are implementation-detail decisions that don't affect the UI/UX mockup (the user-facing behavior is identical regardless: "this looks similar to an existing asset," with a link to it) and can be deferred to Phase 3 without blocking Phase 2.

| *(not in spec)* | `asset_level` (`agency`/`brand`/`project`) | Org-specific, tied to the agency/brand-project hierarchy `ORG_WORKSPACE_SPEC.md` describes; has no personal analog at all, correctly excluded |
| *(not in spec)* | `folder_path`/`folder_id` | **Deliberately excluded from personal per `LIBRARY_SPEC.md` ┬¦2.2** ("No folder hierarchy... structural organization happens entirely through tags/filters") ŌĆö confirms personal's table should *not* mirror this part of org's schema, the spec is explicit about this divergence already |
| *(not in spec)* | `metadata` (jsonb, catch-all) | A flexible escape hatch org uses for `storage_bucket`/`storage_path`/`original_file_name`/`folder_id` (per the `org-asset-upload` edge function's insert payload, `:283-288`) ŌĆö worth carrying forward into personal's table too, even though ┬¦2.1 doesn't name it explicitly, since it's a low-cost, low-risk pattern for exactly the kind of "storage plumbing detail that isn't really user-facing metadata but needs to live somewhere" data the upload edge function will also need to record (storage bucket/path, original filename) |

### 3.2 Net recommendation: structurally mirror `org_asset_library` where the underlying concept is shared, diverge where personal's spec is explicit about a different need ŌĆö not a uniform "copy it" or "start over"

This is not a one-line answer because the evidence doesn't support one. Specifically:

**Mirror directly (same column, same type, same semantics):** `id`, `uploaded_by` (personal: rename conceptually to fit alongside `user_id`, see below), `name`/`title`, `description`, `file_url`/`thumbnail_url` pattern, `file_type`, `mime_type`, `file_size_bytes`, `dimensions`, `tags`, `is_archived`, `created_at`/`updated_at` + the `set_updated_at()` trigger, and the `metadata` jsonb catch-all.

**Adapt the scoping columns to personal's existing convention, not org's:** replace `organization_id`/`brand_project_id`/`asset_level` with `user_id` + `workspace_type` (constrained to `'personal'` for this table, or include `organization_id IS NULL` as a check, consistent with how `PERSONAL_WORKSPACE_SPEC.md` ┬¦1 describes every other personal table being scoped) ŌĆö this is the audit's own flagged open item (AS_IS_AUDIT.md ┬¦7.1, the org-draft-creates-a-stray-personal-row issue) resolved at the schema level: if the new table's RLS/insert policy requires `workspace_type = 'personal'` explicitly, an org-scope write can't land in it by accident the way it currently can with `content_library_items`.

**Add columns org's table doesn't have, because personal's spec requires them and org doesn't:** `source` (`upload`/`generation`/`post`), `generation_id`/`post_id` (direct nullable FKs, not a separate junction table ŌĆö personal's simpler one-asset-to-many-posts relationship via `used_in_post_ids` doesn't need org's `org_post_asset_links` many-to-many-with-role complexity), `alt_text`, `ai_tags`, `checksum` (+ a second column for the perceptual hash specifically, e.g. `perceptual_hash`, distinct from `checksum`'s exact-hash per ┬¦2's two-tier recommendation ŌĆö the spec's singular "checksum" field name undersells that two distinct values are being recommended here, worth flagging for whoever writes the actual migration), `used_in_post_ids` (array, not org's `usage_count` integer), `superseded_by_asset_id` (not org's `versions`/`current_version` jsonb).

**Deliberately omit, per the spec's own explicit instruction:** `folder_path`/`folder_id` (┬¦2.2 ŌĆö tags/filters instead of folders), `approval_status`/`approved_by`/`approved_at`/`rejection_comment` (┬¦3 ŌĆö no approval substate on personal), `is_brand_asset`/`asset_level` (no org-hierarchy concept on personal).

**Reasoning for this mixed approach rather than picking one extreme:** a uniform "copy org's schema exactly" would import folder/approval/brand-hierarchy columns the spec explicitly says personal shouldn't have, and would miss `source`/`ai_tags`/`checksum`/`used_in_post_ids`/`superseded_by_asset_id` entirely, since org's table ŌĆö built for a different set of requirements (approval workflows, folders, agency hierarchy) ŌĆö never needed them. A uniform "design from `LIBRARY_SPEC.md` ┬¦2.1 alone, ignore org" would forfeit the one piece of real, load-bearing precedent this codebase already has for "what does a working asset table with versioning/usage-tracking/audit-trail actually look like running in production here" ŌĆö the `set_updated_at()` trigger convention, the jsonb-`metadata`-catch-all pattern, the `dimensions` jsonb shape, and the general column-naming conventions (`file_size_bytes`, not `fileSize` or `size_bytes`) are all real precedent worth keeping for consistency across the two asset tables that will now exist side by side in the same schema. Treating org's table as "the closest existing working example, adopted selectively" rather than either "the template to clone" or "irrelevant, personal is different" is the position the evidence actually supports.

### 3.3 RLS policy structure ŌĆö also worth mirroring conceptually, adapted to personal's simpler permission model

`org_asset_library`'s RLS policies (`20260324130000` + `20260327020000_org_asset_folders_stage2.sql:452-497`) gate every write through `get_member_permission(organization_id, 'can_manage_library')` and every read through `org_current_user_has_brand_access(...)`. Personal has no equivalent permission system at all (confirmed, AS_IS_AUDIT.md ┬¦2's ┬¦9 finding: "no permission check of any kind... beyond `user_id` ownership," which is correct per `LIBRARY_SPEC.md` ┬¦9's personal row, "always"). The new personal `assets` table's RLS should follow the same simple `user_id = auth.uid()` pattern every other personal table already uses (`media_assets`/`content_library_items`'s existing policies, `20260227090000_calendar_library_alignment.sql:131-147`, confirmed in Phase 0) ŌĆö there is no permission-function layer to port over from org's side, because personal genuinely doesn't need one. This is a place where "mirror org's structure" stops applying, correctly, because the underlying requirement (org's multi-user permission gating) doesn't exist on personal's side at all.

| 3 | `assets` table structure | Structurally mirror `org_asset_library` for shared concepts (id, naming conventions, `metadata` jsonb catch-all, `updated_at` trigger pattern, `dimensions` shape) and personal's own existing scoping convention (`user_id` + `workspace_type='personal'`, not org's `organization_id`/`brand_project_id`/`asset_level`); add `source`/`generation_id`/`post_id`/`alt_text`/`ai_tags`/`checksum`+`perceptual_hash`/`used_in_post_ids`/`superseded_by_asset_id` since org's table has no equivalent and personal's spec requires them; omit `folder_path`, `approval_status`-family, and `is_brand_asset`/`asset_level` since `LIBRARY_SPEC.md` ┬¦2.2/┬¦3 explicitly excludes folders and approval from personal.
