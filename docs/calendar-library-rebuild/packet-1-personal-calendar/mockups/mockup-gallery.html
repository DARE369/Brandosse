Personal Content Calendar — Mockup Gallery

Static, interactive HTML/CSS mirroring real tokens extracted from UserDashboard.jsx, CalendarV3.css, and ui-primitives.css. No backend wiring; representative content only. Every view below is fluid — resize the browser to confirm mobile (~390px) → tablet (~768px) → desktop (~1440px) parity in the same markup, per Master Brief §4. Three distinct reschedule interaction modes are demonstrated, all genuinely clickable.

View

Month view (+ live Month/List default-switching)

Up to 3 grouped cards per day, then "+N more" opens a slide-over list (not a single arbitrary post — spec §3). Platform-icon-stack grouping by generation_id; a post with no generation_id (e.g. a Quick-Post-with-no-asset draft) renders as its own standalone card, never bucketed with other null-generation posts (Research §3.2). Click "+2 more" on Jun 17, or click any card to open the detail drawer. Fix 1 (2026-06-24): this frame now also hosts the Agenda/List panel behind the same Month/List header toggle. Below ~600px frame width it loads to Agenda/List by default (Month is one tap away); resize this frame's right edge across 600px to see the on-load default change. See DECISIONS_LOG.md.

Drag

Desktop mouse drag, and touch drag via @dnd-kit's TouchSensor (already configured in the existing build with { delay: 250, tolerance: 8 } — Research §2). This mockup uses native HTML5 drag to demonstrate the same intent without re-implementing dnd-kit in static HTML. Try it: drag "Q3 recap carousel" onto another day below.

On touch devices, the equivalent is a press-and-hold drag (delay 250ms) via TouchSensor — mechanically functional, but per Research §2 this does not by itself satisfy WCAG 2.2 SC 2.5.7. Mode 3 below is the required non-drag alternative, not just a mobile nicety.

Reschedule mode 2 of 3

Full detail-panel edit

Open the post, change date/time fields directly — already exists today in PostPanel.jsx (native <input type="date"> / <input type="time">, timezone-correct via zonedDateTimeToUTC()), carries forward unchanged into PostDetailDrawer.jsx. This is a real, always-available single-pointer path — heavier than drag, but it satisfies WCAG 2.5.7 on its own merits and is also where caption/platform edits happen.

Open the full drawer to reschedule via date/time fields

Reschedule mode 3 of 3 — new, designed for this packet

Tap-to-select → tap-destination

The genuinely separate, lighter-weight, non-drag single-pointer path Research §2 concluded is required (WCAG 2.2 SC 2.5.7; Master Brief §4) — TouchSensor makes touch-drag work mechanically but does not satisfy this on its own, and the full detail panel (mode 2) is heavier than this needs to be for a simple date move. Works identically at every width — this is not a mobile-only fallback; a desktop mouse user who simply prefers not to drag can use it too.

Before selection, destination cells look identical to any other day (no permanent visual noise). After tapping "Move," every valid destination gains a dashed accent outline + tint (.is-drop-candidate) and the source card gets a matching outline (.is-selected-for-move) — status communicated by shape + color + the sticky banner's text together, never color alone. Escape, the Cancel button, or tapping the same card again all exit the mode without moving anything.

State

Drag conflict

Same account + same platform + same exact timestamp already occupied → non-blocking toast with "schedule anyway," never a hard block (spec §5) — intentional double-posting to the same slot is a real, if rare, use case.

Click to fire the conflict toast

State

Stale write (concurrency conflict)

Every reschedule write includes the row's last-known updated_at; if stale (someone/something else moved it first), the UI rolls back and re-fetches that single card — toast + automatic single-card refresh (spec §5, §10). On personal accounts this typically means a second device/tab, not a teammate.
